# Аудит спеки UI v2 — `docs/superpowers/specs/2026-07-13-synapse-ui-v2-design.md`

Сверка построчно — по номерам S* в тексте спеки.

Это аудит **спеки** (черновик v2.1 «на ре-ревью») — находки чинятся правкой документа, кода v2 ещё нет. Ссылки на код = существующие M1-механизмы, на которые спека опирается.

## CRIT — ломают дизайн, чинить до ре-ревью

- **S1 · Быстрый путь противоречит ядру спеки.** CODE-запуск определён как «реализуй по плану `<файл>`» (§3, стр. 65), но на быстром пути SPEC_PLAN не запускается и план-файла не существует (стр. 66) — задача CODE-запуска неопределима. Заодно: требование 7 («код ТОЛЬКО после явного „ок, пиши код“», стр. 15) на быстром пути нечем удовлетворено — PROPOSE-подтверждение семантически «отправляй запрос», а «CODE-запуск как сейчас» отсылает к fire-and-forget до-v2. И карточки выбора модели для CODE на этом пути нет (карточка PROPOSE скоупнута под модель SPEC_PLAN, стр. 36).
- **S2 · Отказные переходы автомата не определены.** DONE/FAILED нарисованы только после CODE (стр. 49): провал/дедлайн SPEC_PLAN-запуска не ведёт никуда; ре-запуск упавшей стадии не описан (FAILED — тупик без исходящих стрелок); cancel (§4, стр. 73) не назначает стадию; из SPEC_PLAN нет пути «план не нравится, правь» — карточка «План готов» (стр. 37) в отличие от PROPOSE не имеет [Правки], единственный выход — вперёд в CODE. Смежно: переход SPEC_PLAN→CODE не гейтится наличием план-файла («обязан оставить» — модальность без механизма, стр. 59).
- **S3 · Лента кора-шагов не переживает рестарт — требование 11 невыполнимо как написано.** Ring-буфер один на хост, в памяти, 500 записей (`app.py:217`, `config.py:88`); TaskStore хранит ровно одну задачу и затирается каждым `start_task()` (`state.py:192-196`). Спека обещает «история ленты» в персисте треда и `GET .../feed` «включая кора-шаги» (§4, стр. 72, 75-76), но не назначает писателя, который довозит кора-шаги из буфера в персист треда. Рестарт хоста / вытеснение чужим трафиком → шаги старых тредов безвозвратно исчезают.
- **S4 · `POST /api/threads/{id}/gate` без origin-защиты = CSRF на запуск CODE.** Tailnet — сетевая граница, не браузерная: auto-submit `<form>` с любой страницы в браузере tailnet-устройства дёргает gate без preflight (CORS-middleware/auth в коде нет — `webrtc_server.py` целиком). Плюс асимметрия: голосовой деструктив идёт через Р-16 double-key (`confirm.py`), HTTP-кнопка «пиши код» (полный write-доступ к проекту) — нет; и «дублирует голосовые триггеры» неточно — tool перехода стадии в `dispatcher/tools.py` не существует, это новый контур.

## MAJOR

- **S5 · Гейт-эндпоинт недоспецифицирован.** Один POST на три смысла (добро на запрос / пиши код / правки); поля модели нет ни в одном эндпоинте §4, хотя карточки её несут (стр. 36-37, 77); транспорт кнопки [Правки] не назначен (gate или message?).
- **S6 · Гейт не идемпотентен и не сериализован.** `KoraRunner.start()` supersede-cancel'ит живой ран (`kora.py:337-349`) — двойной тап = убитый посреди Write первый запуск + второй, не знающий об этом; гонка голос+UI: `query()` уходит до identity-check (`kora.py:396-402`). Спека обязана потребовать single-flight/дедуп гейта.
- **S7 · «Тот же turn-цикл, мимо STT» — не бесплатен.** `_current_turn_id`/дедуп общие с голосовым конвейером без канал-scoping (`dispatcher/tools.py:145-158`, `app.py:309-320`): одновременный voice-turn и HTTP-turn перетирают друг друга. Ровно класс «переиспользование переоценено», который v2 объявила исправленным.
- **S8 · Светофор в спеке неполон против кода.** В коде ещё два пути: `UNREACHABLE → red` и `CANCEL_REQUESTED → yellow` (`webrtc_server.py:64-76`); буквальная реализация по тексту спеки (стр. 27) даст жёлтый вместо красного для недостижимой Коры.
- **S9 · «Cap по $ — отдельная работа» — ложное утверждение.** `kora_max_budget_usd = 1.0` уже передаётся в КАЖДЫЙ запуск Коры (`kora.py:463`, `config.py:81`) и транслируется SDK в `--max-budget-usd`. Call-count-кап (`record_paid_attempt`) — про дешёвый каскад диспетчера, другая система (§7, стр. 101 смешивает их).
- **S10 · Компакт треда: не зафиксировано, что выжимка работает только по диспетчерскому журналу.** Компакт «истории ленты» целиком прогнал бы сырые кора-шаги через LLM диспетчера — нарушение NO-EXFIL (`kora.py:205-215`: «never … the dispatcher's LLM context»), который §7 объявляет нетронутым. Также не описаны: кто делает выжимку (Р2-риск №3), confirm перед необратимым сжатием (прецедент M1 §2.6).
- **S11 · Голосовое добавление проекта — три места расходятся.** §2.1 (стр. 29) и §4 (стр. 72) описывают его с confirm-читкой как v1-фичу; §6 О3 (стр. 95) — «дефолт v1 = UI-only, голосом позже»; §5 UI-1 перечисляет только UI-добавление.
- **S12 · Клетка произвольного пути проекта без валидации.** Абсолютный путь = что угодно (`/`, `$HOME`); денилист фиксирован (`kora.py:70-83`) и не покрывает `~/.config/gh`, `~/Library/Keychains`, `.zshrc`/`.bash_profile` (запись = persistence). При проекте-$HOME всё это «allow».
- **S13 · Крэш сервера посреди RUNNING блокирует ВСЕ треды навсегда.** Реконсиляции нет (M1.1, живой репро в M1 §2.7): `_load()` восстанавливает статус as-is, liveness врёт OK (`state.py:148-161, 252-253`), `has_active_task()` режет любой submit (`confirm.py:142-146`). Очередь v2 «вежливый отказ» стоит на этом молча — реконсиляцию надо втянуть в v2 или явно принять риск.
- **S14 · Относительные пути PWA-инъекции сломаются на `/client/dev`.** `./session-alive`, `./kora-status`, `./logs` (`webrtc_server.py:47-55`, `reconnect.js:22`, `status-widget.js:24,29`) с dev-страницы резолвятся в `/client/dev/*` → 404. Спека говорит «проверить» — ответ: сломается, нужен реврайт путей или absolute.
- **S15 · «5 стадий» — подсчёт нечестен в обе стороны.** Чип «ГОТОВО·УПАЛ» склеивает DONE и FAILED в одну стадию (стр. 45, 49, 53), а светофор различает FAILED→красный отдельно (стр. 27); enum стадий в реализации обязан их разделять.

## MINOR

- S17 · Текстовый ответ на вопрос Коры: явно прописать, что `POST /message` покрывает и awaiting-путь (диспетчер зовёт `answer_kora`).
- S18 · «Исполняющийся тред» (стр. 73) определить как «есть live-запуск Коры»: COLLECT-тред синглтон не занимает.
- S19 · `KORA_WORKSPACE_DIR` общий для всех project-less тредов — файлы несвязанных «агент»-тредов перемешиваются.

- S21 · Термин «задача» = и запуск (task_id, стр. 71), и весь запрос пользователя (стр. 12, 66) — развести.
- S22 · «Стадия = отдельный запуск Коры» верно для 2 стадий из 5 (COLLECT/PROPOSE без запуска) — переформулировать точнее.
- S23 · Требование 3 «какой тред создавать» vs механика §4 «привязка существующего» — согласовать.
- S24 · Порядок mount'ов: `/client/dev` регистрировать ДО `/client` (Starlette матчит по порядку) — одна строка в §4.
- S25 · `manifest.webmanifest` хардкодит `start_url`/`scope: /client/` — на dev-странице установится не тот ярлык.
- S26 · Вендоринг pipecat JS SDK: версия/пин/лицензия и проверка «ноль сборки» (bare-specifier импорты) не оговорены.
- S27 · iframe-фолбэк Р1 зависит от исправности `/client/dev` — который ломается по S14 ровно тогда, когда фолбэк нужен.
- S28 · `projects.json` — конкурентный read-modify-write (UI+голос) без лока.
- S29 · COLLECT-переспрашивание жжёт платный каскад/RPM — расширить Р2 третьей точкой (объём, не только качество).

- S32 · Лимиты персиста тредов (число/размер файла истории) не заданы — в отличие от `kora_log_max`.
- S33 · Часовой пояс/формат «время» в списке тредов не оговорены.
- S34 · Model-id захардкожены; `kora_model` — bare str без allowlist (`config.py:78`) — HTTP-выбор модели без валидации уйдёт прямо в `ClaudeAgentOptions`.
- S35 · Описание `_gate_decision` как цепочки «секреты → клетка → egress» неточно: код ветвится по типу инструмента (`kora.py:477-506`) — не влияет на вывод спеки, но формулировку поправить.
- S36 · Инвентарь сигналинг-роутов неполон: PATCH `/sessions/{id}/api/offer`, `POST/PATCH /api/offer`, `GET /` (редирект) не упомянуты (`webrtc_server.py:238-254`) — риска для миграции нет, но «полный список» — нет.
