## 1. Что такое Синапс и рамки M0

Синапс — голосовой мост между пользователем и Корой, исполнителем класса Claude Code на домашнем компьютере пользователя. Пользователь голосом (в том числе за рулём) даёт задачи («скачай Дюну, вырежи главу 1 в Markdown»), спрашивает статус, отменяет. Диспетчер — быстрая LLM посередине — ведёт живой разговор, но **не исполняет и не видит работу напрямую**.

**M0 включает:**
- один пользователь, одна активная задача;
- рабочий язык — русский с английскими вкраплениями (code-switching);
- полный голосовой круг: команда голосом → Кора получила; вопрос о статусе → честный ответ; завершение/результат → пользователь слышит дословный текст Коры;
- регресс-контур конфабуляций на проде.

**M0 не включает** (§9): режим надиктовки длинного ТЗ, мультипользовательность, OpenAI Realtime как основной путь, аутентификацию голоса/канала (M0-допущение — push-to-talk на доверенном устройстве, единственный пользователь; §11.1).

## 2. Центральный риск и эшелонированная оборона

Быстрый «глупый» диспетчер под давлением мгновенного ответа выдумывает статус («ну что, готово?» → «да, всё сделано!») или изображает возможности, которых нет. Испытания №1–№2а показали: наивный Haiku врёт в 60% диалогов **при правде перед глазами** — это дефицит дисциплины, не информации; строгий промпт снимает почти всё, но упирается в потолок (whack-a-mole на «утешительных прогнозах»). Поэтому оборона — слоями, а не одним механизмом:

| Слой | Механизм | Что закрывает | Доказано |
|---|---|---|---|
| 1 | Промпт v3 (Приложение А) | конфабуляции и фантомы в светской части | Haiku 61/63 — агрегат трёх версий промпта, на финальном v3 отдельно 23/24; Sonnet 39/39; фантомы 0/27 — агрегат обеих моделей (Haiku 0/18 + Sonnet 0/9) |
| 2 | Grounding: `get_task_status()` обязателен перед любым ответом о ходе задачи (Р-2) | остаточные ~3% Haiku, которые промптом не выбить | размер остатка показан испытаниями; сам механизм tool-call не прогонялся — проверка в №4 |
| 3 | SPEAK: критичные факты (результаты, имена файлов, сроки) — только готовым текстом Коры, дословно через TTS, минуя LLM (Р-5, Р-15) | конфабуляцию критичных фактов | конструкцией гарантирована дословность доставки SPEAK→TTS; запрет пересказа диспетчером — Р-15 (редакция [СОСТОЯНИЕ] + правило 8 + регресс), статус проектный |
| 4 | Регресс `confab_regression.py` при каждой смене модели/промпта + канарейка судьи (Р-3, Р-8) | тихую деградацию со временем | стенд работает с №1 |

Погоня за 100% чистоты промптом прекращена (Р-1): точечная правка убивает формулировку, рефлекс вежливого утешения просачивается в новой.

Честная рамка доказательности (ревью 2026-07-10): все цифры таблицы сняты на *тексте* без ASR при t=1.0, прод — минимальная t с ASR во входе — **две неизмеренные дельты** до прод-репрезентативных чисел (пере-базлайн — §8 крит.4). Слои 2–3 и stale-оборона (Р-11) — проектные механизмы: правила 7–8 в текст промпта ещё не внесены (Прил. А, owed). До прогона №4 таблица — доказанный *потенциал* обороны, не её прод-состояние.

## 3. Архитектура

Каскад STT → LLM → TTS на оркестраторе Pipecat (Р-6). OpenAI Realtime отклонён как основной путь: нижний предел t=0.6, аудио-модели держат инструкции хуже, инструкции деградируют в долгих сессиях, дословность SPEAK не гарантируется; остаётся сравнительным плечом регресс-теста.

```
 микрофон                                                            динамик
    │                                                                   ▲
    ▼            частичные                             стрим            │
 ┌─────────┐    транскрипты    ┌──────────────────┐    текста     ┌─────────┐
 │   STT   │ ────────────────► │  Диспетчер (LLM) │ ────────────► │   TTS   │
 │  (слот) │                   │  (слот) · v3     │               │  (слот) │
 └─────────┘                   └──────────────────┘               └─────────┘
      ▲                           │            ▲                       ▲
      │ Keyterm Prompting         │ tools      │ [СОСТОЯНИЕ]           │ SPEAK: готовый текст
      │ (словарь задачи)          ▼            │ + словарь             │ Коры, дословно, минуя LLM
      │                     ╔═══════════════════════╗                  │
      └─────────────────────║   Мост Коры · стейт   ║──────────────────┘
                            ╚═══════════════════════╝
                                        ▲
                                        │ события задачи · SPEAK-тексты
                           Кора (Claude Code, домашний компьютер)
```

Всё внутри одного Pipecat-пайплайна; turn-taking и barge-in — ответственность оркестратора (и/или STT со встроенной детекцией конца реплики).

### Компоненты

- **Оркестратор — Pipecat 1.x** (v1.0 — апрель 2026, на момент ревизии v1.5; транспорт-агностичен, полный контроль стадий, 60+ интеграций, без lock-in). Запасной — LiveKit Agents, если упрёмся в транспорт/масштаб. Отвечает: аудио-транспорт, VAD/turn-taking, barge-in, маршрутизация SPEAK напрямую в TTS.
- **STT (слот)** — выбирается испытанием №5 (§7). Кандидаты: ElevenLabs Scribe v2 Realtime, Deepgram Flux Multilingual, AssemblyAI Universal-3 Pro Streaming; резерв — self-host Whisper large-v3 (Приложение Б). Главный фильтр — code-switching русский/английский, общий WER на чистом языке его не показывает.
- **Диспетчер (слот) — «быстрая дешёвая LLM», промпт v3, температура минимально возможная (Р-13).** Не фиксируется на одной модели: фильтр слота — цена/скорость + удержание инструкций под давлением + confab-rate на регресс-сетке. Haiku 4.5 — базлайн-кандидат и планка, которую кандидат обязан взять: 96,8% чистых на тексте при t=1.0 — агрегат №1–№2а по трём версиям промпта; на финальном v3 отдельно 23/24 (95,8%); остаток закрывают слои 2–3. Выбор/подтверждение модели — регрессом на M0-пайплайне (Р-3); при недоборе confab-гейта — эскалация на следующую ступень (Sonnet, 100% в испытаниях). Триггер эскалации — измеренный confab-rate, не качество нарратива. **Кандидат M0 и реализация слота (Р-14):** слот — каскад endpoint'ов с автопереключением по 429 / исчерпанию квоты / таймауту **в рамках того же хода**: 1) Google AI Studio `gemini-3.5-flash`, free tier (драйвер выбора — цена; включение billing переводит на платный тир тем же endpoint'ом, без изменения кода) → 2) OpenRouter `google/gemini-3.5-flash` (квота Google исчерпана) → 3) Anthropic Haiku 4.5 — аварийный хвост и единственная ступень вне инфраструктуры Google (ступени 1–2 — одна серверная: OpenRouter проксирует Gemini в тот же Google). Квотный 429 (RPM/RPD-окно — не разовый сбой) приглушает ступень circuit-breaker'ом: она выходит из ротации, последующие ходы идут сразу на следующую ступень без штрафа RTT+retry на каждом ходе. T приглушения зависит от типа окна (из деталей ошибки/retry-delay): RPM-класс — ~60 с; RPD-класс (дневная квота) — до сброса квоты, иначе ступень возвращалась бы каждую минуту в гарантированный 429. Ожидание по деньгам — честное: free tier ступени 1 покрывает редкие/фоновые ходы, в активном диалоге RPM-стена гонит трафик на платные ступени 2–3, так что стоимость сессии ≠ 0 — стоимость/ход меряется в №4 (таблица §11.5); включение billing на ступени 1 снимает RPM-стену и возвращает трафик на Google. Эскалация слота (→ Sonnet) — офлайн-решение по регрессу и судье на прод-журнале; runtime-каскад переключает ступени только по инфра-сигналам (429/таймаут) и качеством не управляет. Переход на ступень 3 — алерт в журнале, голосом не озвучивается (не ошибка для ушей); отказ всех трёх ступеней — голосовая заготовка через SPEAK-путь (§11.3).
- **TTS (слот)** — кандидаты: ElevenLabs Flash v2.5, Cartesia Sonic-3.5, OpenAI gpt-4o-mini-tts, Fish Audio (Приложение Б; референс-клиенты голосового слоя — Приложение Г). Русский — решающий фильтр, качество слушать ушами (smoke-test, §7). Deepgram Aura-2 исключён — русского нет (но его WebSocket-паттерн взят в референс, Прил. Г).
- **Мост Коры / стейт** — принимает поток событий от Коры, ведёт состояние задачи, отдаёт [СОСТОЯНИЕ] и `get_task_status()`, транслирует SPEAK-тексты в TTS-очередь, отдаёт словарь активной задачи обоим слоям.
- **Наблюдаемость** — текст посередине: журнал каждого хода (финальный транскрипт, вход/выход LLM, tool-вызовы, текст в TTS, обслужившая ступень слота — endpoint и модель — и маска приглушённых ступеней на момент хода (состояние circuit-breaker'а), Р-14). Без поля ступени confab-rate и латентность на проде неатрибутируемы: модель может смениться посреди сессии, и «Gemini испортился или весь день сидели на хвосте-Haiku?» останется без ответа. Судья и `confab_regression.py` работают на проде без переделки.

## 4. Контракты и потоки данных

**[СОСТОЯНИЕ]** — единственный источник правды о задаче для диспетчера. Содержит: id задачи, формулировку, время старта, `last_event_ts` (метка последнего события Коры) и полный список событий (`task_started`, прогресс-события Коры, `task_completed`/`task_failed`); каждое событие несёт класс `critical | narratable` (Р-10). Инжектится свежим в каждый ход. **Редакция critical (Р-15):** в [СОСТОЯНИЕ] диспетчера (и в снимок `get_task_status()`) `critical`-события входят редактированными — тип события, метка времени и пометка «детали озвучивает Кора дословно» (обещание, не факт озвучки); payload (имена файлов, результаты, сроки) скрыт. Так запрет пересказа обеспечивается конструктивно: правило 2 промпта запрещает утверждать факты, которых нет в [СОСТОЯНИЕ], — и их там действительно нет. Событие без проставленного класса трактуется как `critical` (fail-safe).

**Живость Коры (Р-11).** Мост держит heartbeat Коры; если `now - last_event_ts` превышает порог устаревания или heartbeat потерян — [СОСТОЯНИЕ] помечается `stale`/`unreachable`. Без этого упавшая Кора неотличима от работающей: стейт замирает на `task_started`, и правило 3 заставляет диспетчера бесконечно и уверенно говорить «сигнала о завершении пока не было» для задачи, которая никогда не завершится, — ровно то утешительное враньё, против которого построена оборона §2. На `stale`/`unreachable` диспетчер обязан сказать правду о потере связи (канон-фраза «давно нет сигнала от Коры, не знаю её состояние»; правило 7 промпта — owed, перепроверяется регрессом Р-3), а не о ходе задачи.

**Инструменты диспетчера** — ровно четыре, зеркало списка возможностей из промпта (подтверждение — часть возможности «принять задачу»):
- `submit_task(text)` — передать новую задачу; на мосту двухфазна (Р-16): обычная задача коммитится сразу, `requires_confirmation` — переходит в `pending_confirmation`;
- `confirm_task(decision)` — завершить подтверждение деструктива (Р-16);
- `get_task_status()` — свежий снимок состояния; **обязателен в том же ходе перед любым ответом о ходе/результатах задачи** (Р-2);
- `request_cancel()` — передать Коре запрос на отмену.

Механическое правило наблюдаемости: ход, в котором диспетчер говорит о статусе без предшествующего `get_task_status()` в этом же ходе, — алерт в журнале.

**SPEAK и граница critical/narratable (Р-5, Р-10).** Кора отдаёт готовые к озвучке фразы (результаты, имена файлов, сроки). Оркестратор шлёт их в TTS напрямую, минуя диспетчера. Гарантия «дословность по конструкции» покрывает ровно то, что Кора пометила `critical`: диспетчеру **запрещено** озвучивать событие класса `critical` своими словами — его речь ограничена светской частью и `narratable`-событиями (Р-5). Без явной классификации граница была дырой: критичный факт, пришедший прогресс-событием (напр. «глава 1 сохранена в chapter-1.md»), диспетчер пересказывал своими словами и терял имя/время/уверенность. Класс проставляет Кора-продюсер — и это переносит часть гарантии на него: неверная классификация (`critical` помечен как `narratable`) или непригодный к озвучке SPEAK-текст роняют гарантию тихо, а испытания №1–№2а продюсера не проверяли. Поэтому граница держится тремя механизмами Р-15, не промптом: (а) редакция critical в [СОСТОЯНИЕ] — см. выше; (б) fail-safe умолчание «нет класса = critical»; (в) **producer-сетка (owed, пререквизит гейта §8 крит.2)** — эталонный набор событий Коры, на котором проверяются классификация и пригодность SPEAK-текстов, отдельно от 8 провокаций диспетчера; (г) **runtime-инвариант «critical ⇒ парный SPEAK»**: critical-событие, к которому в пороговое окно не пришёл SPEAK-текст, — алерт в журнале (по аналогии с алертом «статус без grounding»); иначе редакция Р-15 превращает пропавший SPEAK в тишину, которую пользователь не услышит, а лог не заметит. В сетку диспетчера добавляется провокация «выуживание» («а как файл называется?») — диспетчер не должен зачитывать критичный факт из словаря задачи, который остаётся в его контексте для распознавания входа (Р-9).

**Арбитраж очереди озвучки (уточнение Р-5).** Очередь TTS одна, у оркестратора, и в неё конкурентно пишут два источника: SPEAK-тексты Коры и стрим диспетчера. Приоритет у SPEAK: он встаёт в голову очереди; текущая фраза диспетчера допевается до границы предложения, недовоспроизведённый хвост светского ответа отбрасывается — по построению Р-15 он не содержит критичных фактов, терять нечего. Barge-in пользователя прерывает оба потока (§10). **Интра-ходовой retry каскада (уточнение Р-14):** если ступень упала *после* начала стрима (обрыв WS, таймаут mid-stream), оркестратор сбрасывает неозвученный хвост этого хода из TTS-очереди и аннулирует частичный текст диспетчера, следующая ступень генерирует ответ заново; уже прозвучавший фрагмент не возвращается. Ход помечается в журнале как retry — иначе обрубок + противоречивое продолжение от второй ступени, и латентность/качество хода неатрибутируемы.

**Подтверждение деструктива (Р-12; протокол вызовов — Р-16).** Классификатор деструктива — на стороне Коры/моста, не на решении быстрого диспетчера. Ревью 2026-07-10 вскрыло разрыв: флаг `requires_confirmation` рождается на мосту *после* сабмита, а read-back нужен *до* передачи Коре — при трёх инструментах у цикла не было завершения. Протокол Р-16 закрывает порядок вызовов:
1. диспетчер вызывает `submit_task(text)`;
2. мост классифицирует: обычная задача — коммит сразу; необратимое (удаление, перезапись, деплой) — статус `pending_confirmation`, Коре не уходит. Read-back мост строит **детерминированно: шаблон + дословный транскрипт задачи** («Подтверди необратимую задачу: “<текст как распознан>”») и шлёт в TTS через SPEAK-путь — NLU/извлечение цели не требуется, подтверждается ровно тот текст, который уйдёт Коре. Классификатор `requires_confirmation` — отдельный механизм Коры/моста: ему LLM не запрещён (его выход — флаг, не речь, в TTS не попадает); сомнение классификатора → флаг ставится (fail-safe);
3. пользователь отвечает голосом;
4. диспетчер вызывает `confirm_task(decision)`; мост принимает подтверждение по **двойному ключу**: (а) между шагами 2 и 4 был ход пользователя — `confirm_task` без промежуточной реплики отклоняется (страховка от самовольного подтверждения LLM; провокация «подтверди сам» в регресс-сетке); (б) транскрипт этого хода проходит детерминированную affirm-проверку самого моста (узкий словарь утверждений — «да», «подтверждаю», «делай»; всё прочее, включая нерелевантную реплику или шум, — не подтверждение → повторный read-back, **максимум два**; после — staged-задача сбрасывается с голосовой заготовкой «подтверждение не разобрал, задача отложена» — петля без потолка за рулём это блок + фрустрация; словарь утверждений пополняется по реальному узусу из журналов №4). Решение LLM и affirm-проверка должны совпасть; расхождение = отказ;
5. отрицание или таймаут → staged-задача сбрасывается.
Это разрывает цепочку «STT ослышался (WER на машинном фоне, §7) или словарь примагнитил термин (Р-9) → необратимое действие → SPEAK дословно подтверждает несанкционированное удаление»; за рулём визуальная проверка недоступна, поэтому подтверждение только голосовое.

**Словарь активной задачи (Р-9)** — один контекстный словарь (имена файлов, форматы, термины задачи), две точки инжекта:
- Keyterm Prompting в STT-запрос (+0 мс — параметр того же запроса); поддержка keyterm-инжекта у конкретного кандидата — критерий №5: у Flux не подтверждена (Keyterm Prompting документирован для классического `/listen`, Flux — другой API, Прил. Г); без неё STT-половина Р-9 отпадает и словарь остаётся только в промпте;
- словарный блок в промпте диспетчера (+100–200 токенов префилла) с правилом: «транскрипт может содержать ошибки распознавания; канонические имена — в словаре задачи; критичные детали пользователю озвучивает Кора» (редакция после Р-15 — раньше правило отсылало к [СОСТОЯНИЕ], где critical-payload больше не лежит; перепроверяется в №5б).

Отдельный LLM-корректор между STT и диспетчером **отклонён по латентности** (+500–900 мс при бюджете ~800 мс на круг). Риск словаря — «примагничивание» терминов, которых пользователь не говорил (новый канал конфабуляции на входе); закрывается канарейками №5б: провал канарейки → словарное правило переписывается.

## 5. Бюджет латентности

Идеал живого разговора — **~800 мс на круг** (конец реплики пользователя → первые звуки ответа); рабочий таргет M0 до замера топологии — **1,2–1,5 с** на сотовой (собственная оценка ниже, owed), потолок-незачёт — 2,0 с (§8 крит.3). Стриминг на каждом стыке: STT шлёт частичные транскрипты, LLM стримит в TTS. **Оговорка (разбор 2026-07-10):** таблица ниже — бюджет *обработки у провайдеров*, он не равен wall-clock «рот→ухо». Реальный путь сериализован через облака: авто-микрофон → сотовая → Синапс (Pipecat) → облако STT → назад → облако диспетчера (LLM) → назад → облако TTS → назад → сотовая → динамик.

| Стадия | Ориентир (обработка) |
|---|---|
| STT: endpointing (детекция конца реплики) | ~300 мс тишины *до* финала — неустранимо |
| STT: финальный транскрипт после конца реплики | ≤ 300–500 мс (замер в №5) |
| Диспетчер: первые токены (реплика без grounding) | ~200–300 мс |
| Диспетчер: **статус-ход (Р-2)** | **два прохода** диспетчера: `tool_call` → результат → генерация речи ≈ **450–650 мс** |
| TTS: TTFA | 40–150 мс |

**Owed — бюджет обязан это учитывать, но пока нет:**
- **Топология хостинга Синапса не задана.** Где крутится Pipecat (VPS / облако / телефон) определяет плечо user↔Синапс — единственное, что мы контролируем; сейчас оно не посчитано.
- **Сетевой RTT по каждой ноге.** На LTE каждое плечо через границу телефона ~50–150 мс, облако-к-облаку добавляет ещё. Для «за рулём» на сотовой реалистичный круг — **1.2–1.5 с, не 800 мс**.
- **Статус-ход стоит два прохода, а считался за один** (Р-2 требует `get_task_status()` до ответа) — и это самый частый ход. Митигация: спекулятивный пре-фетч статуса по VAD-онсету вопроса, чтобы результат был в контексте до генерации.
- **Цена каскада (Р-6).** Отказ от Realtime покупает дословность SPEAK и дисциплину промпта ценой именно этих сериализованных облачных плеч, которые один round-trip Realtime (~300–500 мс) свернул бы; порог оцениваем против выбранной архитектуры.

`get_task_status()` — чтение локального стейта, десятки мс; латентность статус-хода даёт не оно, а второй проход LLM. Любая добавка в хот-пас проверяется против бюджета — прецедент Р-9. **Гейт §8 крит.3 — медиана на боевом пайплайне; целевое число пересматривается после замера топологии и сотового плеча (owed).**

## 6. Регресс-контур

- `confab_regression.py` (полная сетка: 8 провокаций × повторы, два типа нарушений — конфабуляция и фантом; ревью 2026-07-10 добавило строки «выуживание» и «подтверди сам», Р-15/Р-16) гоняется при каждой смене модели, промпта **или температуры** (Р-3): базлайн снят при t=1.0, прод — на минимальной t (Р-13), направление эффекта не измерено, поэтому смена t = смена конфигурации.
- В M0 добавляется сравнительное плечо gpt-4o-realtime-mini — тот же сценарий против realtime-канала (Р-3, Р-6).
- **Каскад слота (Р-14):** полная сетка — на ступени 1 (Gemini 3.5 Flash, Google direct). Ступень 2 (OpenRouter) — smoke-прогон той же сетки: модель та же, но Р-3 распространяется и на смену serving-пути. Ступень 3 (Haiku) — уже промеренный базлайн. Судья — Claude: для Gemini-строк перекрёстное семейство получается автоматически (Р-8).
- **Канарейка судьи (Р-8):** в каждую сетку подмешивается заведомо грязный диалог; судья его не пометил → прогон невалиден. Судья — Claude, поэтому семейный биас касается **всех Claude-строк, включая Haiku-базлайн**, а не только Sonnet (ревью 2026-07-10: базлайн, по которому проходит/проваливается кандидат, был вне перекрёстной защиты): подвыборка каждой Claude-строки пере-судится судьёй другого семейства (например Gemini Pro); расхождение выше порога → пере-судить сетку целиком. Owed до №4: выбор контр-судьи, численный порог расхождения и процедура разрешения (у контр-судьи свой биас — при массовом расхождении tie-break третьим семейством + ручной разбор N примеров).
- **Воспроизводимость и замер на голосе (owed, разбор):** 8 провокаций, рубрику судьи и определения clean/confab/phantom — вшить/запинить в репо (сейчас `confab_regression.py` вне репо → цифры из спеки невоспроизводимы). Заглавные 96,8% меряны на *тексте* без STT-ошибок во входе; пере-базлайнить на M0-пайплайне с STT в контуре до использования как гейта (§8 крит.4). Версии моделей испытаний (в протоколе фигурирует «Sonnet 4.6» как судья/канал — вне протокола не подтверждается) выверить по скрипту при вшивании; судья и эскалация в M0 — актуальный Sonnet (на момент ревизии — Sonnet 5).

## 7. План валидации

| Испытание | Что проверяет | Гейт |
|---|---|---|
| **№4 «нарратив»** | поток событий Claude Code → живой рассказ через диспетчер (текущий текстовый стенд) | эскалация на след. ступень слота (Sonnet) по **измеренному confab-rate** на M0-пайплайне, не по качеству нарратива (Р-13) |
| **№5 «STT-бейк-офф»** (Р-7) | 30–50 реальных голосовых команд, боевой микрофон, три фона (тихая комната / улица / машина); метрики: WER общий, WER по ключевым терминам отдельно, латентность до финального транскрипта | выбор STT-слота по цена/качество; решаем только на своём аудио — продакшн не совпадает ни с одним бенчмарком |
| **№5б «словарь в двух слоях»** (Р-9) | тот же материал + канарейки 41–45: термины, которых в записях нет, — диспетчер не должен их «примагничивать» | провал → словарное правило переписывается |
| **Smoke-test TTS** (экс-№2) | произношение имён файлов, путей, чисел на русском кандидатами TTS | выбор TTS-слота; дословность SPEAK отдельной проверки не требует — её даёт архитектура |

**Численные гейты (owed — числа проставить до №5):** STT-слот проходит только при keyterm-WER ≤ X% **и** латентности до финала ≤ Y мс (оба порога); TTS — MOS/рубрика ≥ порога на ≥ N слушателей, не «ушами» одного; диспетчер/«нарратив» — балл судьи ≥ порога **и** confab-rate ≤ гейта. Без чисел кандидата нельзя объективно провалить.

Примечание к нумерации: №3 не занят — содержания за ним нет (историческая дыра серии №1, №2, №2а → №4); экс-№2 «SPEAK дословно» переквалифицировано в smoke-test TTS.

**Пререквизиты №5:** записи 30–50 команд (список фраз готовится вместе со скриптом), API-ключи ElevenLabs и AssemblyAI (Deepgram есть — $200 кредитов: по ставке Flux ~$0.008/мин это ≈ 415 ч стриминга, с запасом на базлайн и прототип; прежняя оценка «750 ч» со ставкой не билась — эррата протокола), подтверждение языковой смеси (чисто английский режим → состав кандидатов пересматривается).

## 8. Критерии готовности M0

1. Голосовой круг работает на всех трёх сценариях диспетчера (новая задача, статус, отмена) — на фиксированном корпусе ≥ N диалогов на сценарий (N — owed).
2. Результаты и имена файлов пользователь слышит только через SPEAK — на том же корпусе в журнале нет ходов, где диспетчер пересказывает критичные факты.
3. Медианная латентность диалогового хода ≤ целевого порога, замерена на боевом пайплайне (порог — после замера топологии/сотового плеча, §5). До замера действует страховочный потолок-незачёт: медиана > 2,0 с — провал M0 независимо от результатов замера; иначе гейт нефальсифицируем.
4. Регресс зелёный **на M0-пайплайне с ASR в контуре** (не только на тексте): выбранная модель слота **не хуже Haiku-базлайна на идентичной сетке и версии промпта** (на v3 это 23/24; агрегатные 96,8% смешивают три версии промпта и планкой не являются), фантомы 0, канарейка судьи срабатывает, realtime-плечо отрабатывает как сравнение. Пререквизит — стенд `confab_regression.py` вшит в репо (§6), иначе гейт технически неприменим.
5. Алерт «статус без grounding» за фиксированное окно наблюдения (≥ W часов/диалогов, W — owed) — ноль срабатываний.

## 9. Вне рамок M0

- **Полный Typeless-rewrite** (LLM-полировка транскрипта) — только как отдельный режим надиктовки длинного ТЗ для Коры, вне хот-паса диалога.
- **Погоня за 100% промптом** — закрыта Р-1, потолок показан эмпирически.
- **OpenAI Realtime как основной путь** — только сравнительное плечо регресса.
- Мультипользовательность, просодика входа (для диспетчера некритична), несколько параллельных задач.
- **Аутентификация user↔Синапс↔Кора и authz исполнения** — вне M0 (§11.1). M0-допущение: единственный пользователь, активация push-to-talk на доверенном устройстве. Wake-word / enrolled-voice / mutual-auth канала — отдельный этап.

## 11. Безопасность и деградация (скелет — разбор 2026-07-10)

Добавлено по разбору: §1–§10 виртуозно обороняют один риск (конфабуляция диспетчера), но риски с большим blast radius не покрыты. Здесь — периметр; детальный дизайн частью owed / выносится в отдельный этап.

**11.1 Авторизация и активация (owed; вне M0 — но явно, не молча).**
- **user → Синапс.** Сейчас любой голос в салоне (пассажир, радио, голосовуха) = команда домашнему ПК. M0-допущение фиксируется явно: push-to-talk / привязка к доверенному устройству; wake-word + enrolled-voice — отдельный этап.
- **Синапс → Кора.** Канал моста и то, что Коре *позволено* исполнять, не описаны. Требование: mutual-auth + шифрование канала; authz-политика (allowlist / capability scope), принудительно **на стороне Коры**, независимо от промпта диспетчера.

**11.2 Подтверждение деструктива.** Механизм — Р-12, протокол вызовов — Р-16 (§4): необратимые действия проходят голосовой read-back+подтверждение до исполнения. Разрывает компаундинг-цепочку: STT-ослышка (§7) → нет переспроса → необратимое действие → SPEAK подтверждает несанкционированное; за рулём глазами не проверить. **Компаунд с 11.1 (M0, явно):** без voice-auth подтвердить деструктив может любой голос в салоне (пассажир, радио) — affirm-проверка Р-16 фильтрует семантику, но не личность; порог M0 держится на push-to-talk-допущении, полный фикс — enrolled-voice вне M0.

**11.3 Деградация и фолбэки (owed; LLM-нога закрыта Р-14).** 3+ вендора (STT + TTS + LLM-диспетчер) на критпути одной реплики; за рулём на дёрганой сотовой обрыв WS посреди хода — норма, не край. Требования:
- пер-нога политика отказа: STT reconnect с реплеем аудио; **вторичный TTS авто-своп**; LLM — каскад ступеней Р-14 (429/таймаут → следующая ступень в том же ходе); отказ всех трёх ступеней → голосовая заготовка «связь с мозгом потеряна» через SPEAK-путь — готовый текст напрямую в TTS, LLM для этой фразы не нужен;
- **ошибки доходят до ушей, а не только до лога**: каждый класс отказа → голосовая фраза («связь потеряна», «не расслышал, повтори»); сейчас единственный сигнал — лог-алерт «статус без grounding» (§4), который пользователь не слышит;
- таблица стоимость/ход по всем вендорам до заморозки слотов (Fish — «уточнить», Прил. Б).

**11.4 Приватность (owed).** Сырой голос + транскрипты + содержание задач уходят в сторонние облака (Deepgram, ElevenLabs, AssemblyAI, OpenAI, Anthropic, Google, OpenRouter); retention / consent / redaction не заданы, журнал (§3) хранит полные транскрипты без лимита. Требуется: per-vendor no-train/retention-позиция, retention+redaction журнала, позиция по чувствительному контенту (в салоне вслух — имена, адреса, пароли).

**M0-допущение по free tier (Р-14, зафиксировано явно — как push-to-talk в §11.1):** диспетчер работает на free tier Google/OpenRouter, чьи условия допускают использование данных ходов (транскрипты реплик + [СОСТОЯНИЕ]) для обучения провайдера. Принято осознанно, драйвер — цена. Выход в любой момент без изменения кода — включить billing у Google (ступень 1 становится платной, no-train paid-условия); полный пересмотр приватности — owed вне M0. Внутри допущения зафиксирован и **дрейф политики по квоте**: у ступеней каскада разные политики данных, и переключение между ними происходит автоматически посреди сессии по состоянию квоты (Р-14), а не решением пользователя; постфактум-атрибуция «какой ход под какой политикой» — поле ступени в журнале (§3).

**11.5 Секреты и стоимость (owed).** Ключи (Deepgram / TTS / LLM-каскад: Google, OpenRouter, Anthropic / Fish) — именованное хранилище + ротация, не «ключ уже есть» вразброс. Cost-cap на сессию/сутки с алертом: застрявший barge-in или доступ к неавторизованному микрофону жгут кредиты у всех метред-вендоров. Действие по достижении cap (проектное, без него cap — мониторинг, не защита): hard-stop платных вызовов каскада; пользователь слышит голосовую заготовку через SPEAK-путь; статус-вопросы мост отвечает детерминированным шаблонным рендером [СОСТОЯНИЕ] без LLM; снятие — только ручное решение пользователя. Таблица стоимость/ход — см. 11.3.

**Внесено вторым проходом разбора (2026-07-10):** #6 (§6 owed + §8 крит.4 на голосе), #7 (Р-13, §3/§7), #11 (§7 числовые гейты), #12 (§9 + §11.1), #13 (Прил. Г), #14 (§8 корпус/окно + §11.5). Открытых пунктов разбора не осталось — дальше боевые замеры и детальный дизайн owed-механизмов.

## Приложение Г — референсные WebSocket-реализации голосового слоя

Две рабочие реализации из нашего стека, покрывающие обе ноги каскада. Берутся как **референс протокола и паттернов**, не как готовый прод-код: оркестратор M0 — Pipecat (Python), а источники — TypeScript/React Native. По своему DESIGN `fishaudio-engine` портируется 1:1 на Python/Swift/RN.

**Источники:**
- **`fishaudio-engine`** — `github.com/Terobyte/fishaudio-engine` (**запинить на commit SHA — owed:** внешний репо дрейфует/исчезает) — платформо-агностичный WebSocket-движок Fish Audio TTS (Node 22+/Bun/Deno, msgpack, PCM16LE), с тестами. Чистый эталон: `common/websocket.ts` (обёртка сокета) + `tts/client.ts` (протокол, backpressure, abort).
- **AskME** (`~/Documents/trae_projects/AskME/AskME-Mobile/src/services/`) — боевые RN-сервисы: `deepgram-service.ts` (STT), `deepgram-streaming-service.ts` (Aura-2 TTS), `fish-audio-streaming-service.ts` (Fish TTS).

**Карта «источник → нога каскада»:**

| Файл | Нога | Endpoint | Аутентификация | Формат провода |
|---|---|---|---|---|
| `deepgram-service.ts` | **STT** | `wss://api.deepgram.com/v1/listen?encoding=linear16&sample_rate=16000&endpointing=300&interim_results=true` | `Sec-WebSocket-Protocol: ['token', <key>]` | вход PCM16 → выход JSON-транскрипты (interim + final) |
| `deepgram-streaming-service.ts` | TTS (Aura-2 — **REJECTED, только паттерн**) | `wss://api.deepgram.com/v1/speak?…&model=aura-2-*` | тот же subprotocol | control JSON (`Speak`/`Flush`/`Close`) + бинарный PCM16 |
| `fish-audio-streaming-service.ts` + `fishaudio-engine` | **TTS** | `wss://api.fish.audio/v1/tts/live?model=s1` | `Authorization: Bearer <key>` (header; RN 0.74+/undici) | **msgpack** обе стороны: `start`/`text`/`flush`/`stop` → `audio`/`finish`/`log`; PCM16LE@44.1k |

**Паттерны, которые забираем в M0:**
1. **Детекция конца реплики на STT** — `endpointing=300` + `interim_results=true` у Deepgram `/listen` снимает внешний VAD и напрямую закрывает риск turn-taking из §10. ⚠️ Это **классический** Deepgram-стриминг (`/listen`) — *другой продукт*, чем кандидат **Flux** (у Flux собственная модель конца реплики и свой API/параметры). Паттерн иллюстрирует подход; при выборе Flux endpoint и настройки сверять по его докам, не по `/listen`.
2. **Auth WebSocket без заголовков** — где клиент не может выставить header (браузер / старый RN), Deepgram использует subprotocol-трюк `['token', key]`; Fish идёт через `Bearer` header. На сервере Pipecat (Python) — обычные заголовки, оба варианта тривиальны.
3. **Artifact-free аудио** — msgpack + сырой PCM16LE, никаких codec-frame границ → нет кликов на стыках чанков (совпадает с §5: стриминг на каждом стыке). Правило: чанки не мутировать и не ре-буферизовать; потребитель владеет playback-буфером.
4. **Стриминговый интерфейс** — `async generator` чанков; у Fish есть форма `stream(asyncIterable<text>)` — аудио генерится параллельно токенам LLM (прямо ложится на «LLM стримит в TTS», §5).
5. **Barge-in и устойчивость** — `AbortSignal` прерывает поток и шлёт `stop` (нужно для перебивания реплики, §3/§10); backpressure по `maxBufferedChunks`; reconnect с exp-backoff; ping-keepalive.

**Несущие строки протокола (инлайн — на случай гниения ссылок):**
- **Fish `tts/live`** (msgpack обе стороны): клиент → `{event:"start", request:{format:"pcm", sample_rate:44100, reference_id, latency:"balanced", chunk_length:100}}`, затем N× `{event:"text", text}`, опц. `{event:"flush"}`, финал `{event:"stop"}`; сервер → `{event:"audio", audio:<bytes>}` ×N, `{event:"finish", reason}`, `{event:"log"}`. PCM16LE@44.1k: 2 байта = сэмпл, чанки конкатенируются без кликов, не мутировать. (`latency:"balanced"` — более быстрый TTFB, чем `normal`.)
- **Deepgram `/listen`** (STT): вход — бинарный PCM16; выход — JSON `{is_final, channel.alternatives[0].transcript}`; `endpointing=300` → событие конца реплики.

**Честные оговорки:**
- **Aura-2 TTS (`/v1/speak`) — только паттерн, не провайдер.** Спека уже отклонила Aura-2: русского нет (§3, Прил. Б). Берём схему auth/стриминга, не сам движок.
- Deepgram в AskME присутствует **и как STT, и как TTS** — это два разных файла и два endpoint (`/listen` vs `/speak`); в M0 полезен именно STT-путь.
- **Fish Audio → новый TTS-кандидат** (внесён в Прил. Б): русский у нас уже боевой (`tg.py`, голос Гаврилова), клиент artifact-free и готов. Решающая проверка — smoke-test русского ушами (§7) и замер TTFA. Это кандидат, не принятое решение — отдельного Р-решения пока нет.

**Ссылки на протоколы:** Deepgram STT streaming — `developers.deepgram.com/docs`; Fish Audio WebSocket — `docs.fish.audio/api-reference/endpoint/websocket/tts-live`.
