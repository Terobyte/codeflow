# Синапс · Спецификация M0 — голосовой слой (каскад STT → LLM → TTS)

**Дата:** 2026-07-08 (ревизия 2026-07-10) · **Статус:** принята с условиями — CRIT-находки разбора (§4 Р-10…Р-12, §5, §11) блокируют до закрытия · **Основание:** протокол испытаний №1–№2а (`docs/2026-07-08-synapse-trials-protocol.md`), решения Р-1…Р-13

---

## 1. Что такое Синапс и рамки M0

Синапс — голосовой мост между пользователем и Корой, исполнителем класса Claude Code на домашнем компьютере пользователя. Пользователь голосом (в том числе за рулём) даёт задачи («скачай Дюну, вырежи главу 1 в Markdown»), спрашивает статус, отменяет. Диспетчер — быстрая LLM посередине — ведёт живой разговор, но **не исполняет и не видит работу напрямую**.

**M0 включает:**
- один пользователь, одна активная задача;
- рабочий язык — русский с английскими вкраплениями (code-switching);
- полный голосовой круг: команда голосом → Кора получила; вопрос о статусе → честный ответ; завершение/результат → пользователь слышит дословный текст Коры;
- регресс-контур конфабуляций на проде.

**M0 не включает** (§9): режим надиктовки длинного ТЗ, мультипользовательность, OpenAI Realtime как основной путь, аутентификацию голоса/канала (M0-допущение — push-to-talk на доверенном устройстве, единственный пользователь; §11.1).

## 2. Центральный риск и эшелонированная оборона

Быстрый «глупый» диспетчер под давлением мгновенного ответа выдумывает статус («ну что, готово?» → «да, всё сделано!») или изображает возможности, которых нет. Испытания №1–№2а показали: наивный Haiku врёт в 60% диалогов **при правде перед глазами** — это дефицит дисциплины, не информации; строгий промпт снимает почти всё, но упирается в потолок (whack-a-mole на «утешительных прогнозах»). Поэтому оборона — слоями, а не одним механизмом:

| Слой | Механизм | Что закрывает | Доказано |
|---|---|---|---|
| 1 | Промпт v3 (Приложение А) | конфабуляции и фантомы в светской части | Haiku 61/63 (96,8%), Sonnet 39/39, фантомы 0/27 |
| 2 | Grounding: `get_task_status()` обязателен перед любым ответом о ходе задачи (Р-2) | остаточные ~3% Haiku, которые промптом не выбить | принцип подтверждён испытаниями |
| 3 | SPEAK: критичные факты (результаты, имена файлов, сроки) — только готовым текстом Коры, дословно через TTS, минуя LLM (Р-5) | конфабуляцию критичных фактов **по конструкции** | дословность гарантирована архитектурой каскада |
| 4 | Регресс `confab_regression.py` при каждой смене модели/промпта + канарейка судьи (Р-3, Р-8) | тихую деградацию со временем | стенд работает с №1 |

Погоня за 100% чистоты промптом прекращена (Р-1): точечная правка убивает формулировку, рефлекс вежливого утешения просачивается в новой.

## 3. Архитектура

Каскад STT → LLM → TTS на оркестраторе Pipecat (Р-6). OpenAI Realtime отклонён как основной путь: нижний предел t=0.6, аудио-модели держат инструкции хуже, инструкции деградируют в долгих сессиях, дословность SPEAK не гарантируется; остаётся сравнительным плечом регресс-теста.

```
 микрофон                                                            динамик
    │                                                                   ▲
    ▼            частичные                             стрим            │
 ┌─────────┐    транскрипты    ┌──────────────────┐    текста     ┌─────────┐
 │   STT   │ ────────────────► │  Диспетчер (LLM) │ ────────────► │   TTS   │
 │  (слот) │                   │  (слот) · v3     │               │  (слот) │
 └─────────┘                   └──────────────────┘               └─────────┘
      ▲                           │            ▲                       ▲
      │ Keyterm Prompting         │ tools      │ [СОСТОЯНИЕ]           │ SPEAK: готовый текст
      │ (словарь задачи)          ▼            │ + словарь             │ Коры, дословно, минуя LLM
      │                     ╔═══════════════════════╗                  │
      └─────────────────────║   Мост Коры · стейт   ║──────────────────┘
                            ╚═══════════════════════╝
                                        ▲
                                        │ события задачи · SPEAK-тексты
                           Кора (Claude Code, домашний компьютер)
```

Всё внутри одного Pipecat-пайплайна; turn-taking и barge-in — ответственность оркестратора (и/или STT со встроенной детекцией конца реплики).

### Компоненты

- **Оркестратор — Pipecat v1.0** (транспорт-агностичен, полный контроль стадий, 60+ интеграций, без lock-in). Запасной — LiveKit Agents, если упрёмся в транспорт/масштаб. Отвечает: аудио-транспорт, VAD/turn-taking, barge-in, маршрутизация SPEAK напрямую в TTS.
- **STT (слот)** — выбирается испытанием №5 (§7). Кандидаты: ElevenLabs Scribe v2 Realtime, Deepgram Flux Multilingual, AssemblyAI Universal-3 Pro Streaming; резерв — self-host Whisper large-v3 (Приложение Б). Главный фильтр — code-switching русский/английский, общий WER на чистом языке его не показывает.
- **Диспетчер (слот) — «быстрая дешёвая LLM», промпт v3, температура минимально возможная (Р-13).** Не фиксируется на одной модели: фильтр слота — цена/скорость + удержание инструкций под давлением + confab-rate на регресс-сетке. Haiku 4.5 — базлайн-кандидат (96,8% чистых на тексте при t=1.0; остаток закрывают слои 2–3) и планка, которую кандидат обязан взять. Выбор/подтверждение модели — регрессом на M0-пайплайне (Р-3); при недоборе confab-гейта — эскалация на следующую ступень (Sonnet, 100% в испытаниях). Триггер эскалации — измеренный confab-rate, не качество нарратива.
- **TTS (слот)** — кандидаты: ElevenLabs Flash v2.5, Cartesia Sonic-3.5, OpenAI gpt-4o-mini-tts, Fish Audio (Приложение Б; референс-клиенты голосового слоя — Приложение Г). Русский — решающий фильтр, качество слушать ушами (smoke-test, §7). Deepgram Aura-2 исключён — русского нет (но его WebSocket-паттерн взят в референс, Прил. Г).
- **Мост Коры / стейт** — принимает поток событий от Коры, ведёт состояние задачи, отдаёт [СОСТОЯНИЕ] и `get_task_status()`, транслирует SPEAK-тексты в TTS-очередь, отдаёт словарь активной задачи обоим слоям.
- **Наблюдаемость** — текст посередине: журнал каждого хода (финальный транскрипт, вход/выход LLM, tool-вызовы, текст в TTS). Судья и `confab_regression.py` работают на проде без переделки.

## 4. Контракты и потоки данных

**[СОСТОЯНИЕ]** — единственный источник правды о задаче для диспетчера. Содержит: id задачи, формулировку, время старта, `last_event_ts` (метка последнего события Коры) и полный список событий (`task_started`, прогресс-события Коры, `task_completed`/`task_failed`); каждое событие несёт класс `critical | narratable` (Р-10). Инжектится свежим в каждый ход.

**Живость Коры (Р-11).** Мост держит heartbeat Коры; если `now - last_event_ts` превышает порог устаревания или heartbeat потерян — [СОСТОЯНИЕ] помечается `stale`/`unreachable`. Без этого упавшая Кора неотличима от работающей: стейт замирает на `task_started`, и правило 3 заставляет диспетчера бесконечно и уверенно говорить «сигнала о завершении пока не было» для задачи, которая никогда не завершится, — ровно то утешительное враньё, против которого построена оборона §2. На `stale`/`unreachable` диспетчер обязан сказать правду о потере связи (канон-фраза «давно нет сигнала от Коры, не знаю её состояние»; правило 7 промпта — owed, перепроверяется регрессом Р-3), а не о ходе задачи.

**Инструменты диспетчера** — ровно три, зеркало списка возможностей из промпта:
- `submit_task(text)` — передать новую задачу Коре;
- `get_task_status()` — свежий снимок состояния; **обязателен в том же ходе перед любым ответом о ходе/результатах задачи** (Р-2);
- `request_cancel()` — передать Коре запрос на отмену.

Механическое правило наблюдаемости: ход, в котором диспетчер говорит о статусе без предшествующего `get_task_status()` в этом же ходе, — алерт в журнале.

**SPEAK и граница critical/narratable (Р-5, Р-10).** Кора отдаёт готовые к озвучке фразы (результаты, имена файлов, сроки). Оркестратор шлёт их в TTS напрямую, минуя диспетчера. Гарантия «дословность по конструкции» покрывает ровно то, что Кора пометила `critical`: диспетчеру **запрещено** озвучивать событие класса `critical` своими словами — его речь ограничена светской частью и `narratable`-событиями (Р-5). Без явной классификации граница была дырой: критичный факт, пришедший прогресс-событием (напр. «глава 1 сохранена в chapter-1.md»), диспетчер пересказывал своими словами и терял имя/время/уверенность. Класс проставляет Кора-продюсер; граница проверяется в регресс-сетке (Р-3), а не только промптом.

**Подтверждение деструктива (Р-12).** Задачам с флагом `requires_confirmation` (необратимое — удаление, перезапись, деплой) `submit_task` не коммитит сразу: диспетчер обязан голосом зачитать распознанное намерение (read-back через SPEAK) и получить утвердительный ответ до передачи Коре. Это разрывает цепочку «STT ослышался (WER на машинном фоне, §7) или словарь примагнитил термин (Р-9) → необратимое действие → SPEAK дословно подтверждает несанкционированное удаление». Классификатор деструктива — на стороне Коры/моста, не на решении быстрого диспетчера; за рулём визуальная проверка недоступна, поэтому подтверждение только голосовое.

**Словарь активной задачи (Р-9)** — один контекстный словарь (имена файлов, форматы, термины задачи), две точки инжекта:
- Keyterm Prompting в STT-запрос (+0 мс — параметр того же запроса);
- словарный блок в промпте диспетчера (+100–200 токенов префилла) с правилом: «транскрипт может содержать ошибки распознавания; канонические имена — в [СОСТОЯНИЕ]».

Отдельный LLM-корректор между STT и диспетчером **отклонён по латентности** (+500–900 мс при бюджете ~800 мс на круг). Риск словаря — «примагничивание» терминов, которых пользователь не говорил (новый канал конфабуляции на входе); закрывается канарейками №5б: провал канарейки → словарное правило переписывается.

## 5. Бюджет латентности

Порог живого разговора — **~800 мс на круг** (конец реплики пользователя → первые звуки ответа). Стриминг на каждом стыке: STT шлёт частичные транскрипты, LLM стримит в TTS. **Оговорка (разбор 2026-07-10):** таблица ниже — бюджет *обработки у провайдеров*, он не равен wall-clock «рот→ухо». Реальный путь сериализован через облака: авто-микрофон → сотовая → Синапс (Pipecat) → облако STT → назад → облако диспетчера (LLM) → назад → облако TTS → назад → сотовая → динамик.

| Стадия | Ориентир (обработка) |
|---|---|
| STT: endpointing (детекция конца реплики) | ~300 мс тишины *до* финала — неустранимо |
| STT: финальный транскрипт после конца реплики | ≤ 300–500 мс (замер в №5) |
| Диспетчер: первые токены (реплика без grounding) | ~200–300 мс |
| Диспетчер: **статус-ход (Р-2)** | **два прохода** диспетчера: `tool_call` → результат → генерация речи ≈ **450–650 мс** |
| TTS: TTFA | 40–150 мс |

**Owed — бюджет обязан это учитывать, но пока нет:**
- **Топология хостинга Синапса не задана.** Где крутится Pipecat (VPS / облако / телефон) определяет плечо user↔Синапс — единственное, что мы контролируем; сейчас оно не посчитано.
- **Сетевой RTT по каждой ноге.** На LTE каждое плечо через границу телефона ~50–150 мс, облако-к-облаку добавляет ещё. Для «за рулём» на сотовой реалистичный круг — **1.2–1.5 с, не 800 мс**.
- **Статус-ход стоит два прохода, а считался за один** (Р-2 требует `get_task_status()` до ответа) — и это самый частый ход. Митигация: спекулятивный пре-фетч статуса по VAD-онсету вопроса, чтобы результат был в контексте до генерации.
- **Цена каскада (Р-6).** Отказ от Realtime покупает дословность SPEAK и дисциплину промпта ценой именно этих сериализованных облачных плеч, которые один round-trip Realtime (~300–500 мс) свернул бы; порог оцениваем против выбранной архитектуры.

`get_task_status()` — чтение локального стейта, десятки мс; латентность статус-хода даёт не оно, а второй проход LLM. Любая добавка в хот-пас проверяется против бюджета — прецедент Р-9. **Гейт §8 крит.3 — медиана на боевом пайплайне; целевое число пересматривается после замера топологии и сотового плеча (owed).**

## 6. Регресс-контур

- `confab_regression.py` (полная сетка: 8 провокаций × повторы, два типа нарушений — конфабуляция и фантом) гоняется при каждой смене модели или промпта (Р-3).
- В M0 добавляется сравнительное плечо gpt-4o-realtime-mini — тот же сценарий против realtime-канала (Р-3, Р-6).
- **Канарейка судьи (Р-8):** в каждую сетку подмешивается заведомо грязный диалог; судья его не пометил → прогон невалиден. Для строк Sonnet — перекрёстная проверка судьёй другого семейства (Sonnet судит Sonnet → возможен семейный биас).
- **Воспроизводимость и замер на голосе (owed, разбор):** 8 провокаций, рубрику судьи и определения clean/confab/phantom — вшить/запинить в репо (сейчас `confab_regression.py` вне репо → цифры из спеки невоспроизводимы). Заглавные 96,8% меряны на *тексте* без STT-ошибок во входе; пере-базлайнить на M0-пайплайне с STT в контуре до использования как гейта (§8 крит.4).

## 7. План валидации

| Испытание | Что проверяет | Гейт |
|---|---|---|
| **№4 «нарратив»** | поток событий Claude Code → живой рассказ через диспетчер (текущий текстовый стенд) | эскалация на след. ступень слота (Sonnet) по **измеренному confab-rate** на M0-пайплайне, не по качеству нарратива (Р-13) |
| **№5 «STT-бейк-офф»** (Р-7) | 30–50 реальных голосовых команд, боевой микрофон, три фона (тихая комната / улица / машина); метрики: WER общий, WER по ключевым терминам отдельно, латентность до финального транскрипта | выбор STT-слота по цена/качество; решаем только на своём аудио — продакшн не совпадает ни с одним бенчмарком |
| **№5б «словарь в двух слоях»** (Р-9) | тот же материал + канарейки 41–45: термины, которых в записях нет, — диспетчер не должен их «примагничивать» | провал → словарное правило переписывается |
| **Smoke-test TTS** (экс-№2) | произношение имён файлов, путей, чисел на русском кандидатами TTS | выбор TTS-слота; дословность SPEAK отдельной проверки не требует — её даёт архитектура |

**Численные гейты (owed — числа проставить до №5):** STT-слот проходит только при keyterm-WER ≤ X% **и** латентности до финала ≤ Y мс (оба порога); TTS — MOS/рубрика ≥ порога на ≥ N слушателей, не «ушами» одного; диспетчер/«нарратив» — балл судьи ≥ порога **и** confab-rate ≤ гейта. Без чисел кандидата нельзя объективно провалить.

**Пререквизиты №5:** записи 30–50 команд (список фраз готовится вместе со скриптом), API-ключи ElevenLabs и AssemblyAI (Deepgram есть — $200 кредитов ≈ 750 ч на базлайн и прототип), подтверждение языковой смеси (чисто английский режим → состав кандидатов пересматривается).

## 8. Критерии готовности M0

1. Голосовой круг работает на всех трёх сценариях диспетчера (новая задача, статус, отмена) — на фиксированном корпусе ≥ N диалогов на сценарий (N — owed).
2. Результаты и имена файлов пользователь слышит только через SPEAK — на том же корпусе в журнале нет ходов, где диспетчер пересказывает критичные факты.
3. Медианная латентность диалогового хода ≤ целевого порога, замерена на боевом пайплайне (порог — после замера топологии/сотового плеча, §5).
4. Регресс зелёный **на M0-пайплайне с ASR в контуре** (не только на тексте): выбранная модель слота ≥ базлайна (≥ 96% чистых, фантомы 0), канарейка судьи срабатывает, realtime-плечо отрабатывает как сравнение.
5. Алерт «статус без grounding» за фиксированное окно наблюдения (≥ W часов/диалогов, W — owed) — ноль срабатываний.

## 9. Вне рамок M0

- **Полный Typeless-rewrite** (LLM-полировка транскрипта) — только как отдельный режим надиктовки длинного ТЗ для Коры, вне хот-паса диалога.
- **Погоня за 100% промптом** — закрыта Р-1, потолок показан эмпирически.
- **OpenAI Realtime как основной путь** — только сравнительное плечо регресса.
- Мультипользовательность, просодика входа (для диспетчера некритична), несколько параллельных задач.
- **Аутентификация user↔Синапс↔Кора и authz исполнения** — вне M0 (§11.1). M0-допущение: единственный пользователь, активация push-to-talk на доверенном устройстве. Wake-word / enrolled-voice / mutual-auth канала — отдельный этап.

## 10. Риски

| Риск | Митигация |
|---|---|
| Turn-taking / barge-in — теперь наша проблема (цена каскада) | STT со встроенной детекцией конца реплики (Flux — кандидат) и/или VAD оркестратора; проверка в №5 по интегральной задержке |
| Словарный магнит — конфабуляция на входе | канарейки №5б |
| Русский TTS — акцент/качество | smoke-test ушами до выбора слота |
| Цифры испытаний получены на текстовых Claude, N=15–24 на ячейку | достаточно для разрыва 60% / ~3% / 0%; финальная проверка регрессом на боевом пайплайне M0 |

## 11. Безопасность и деградация (скелет — разбор 2026-07-10)

Добавлено по разбору: §1–§10 виртуозно обороняют один риск (конфабуляция диспетчера), но риски с большим blast radius не покрыты. Здесь — периметр; детальный дизайн частью owed / выносится в отдельный этап.

**11.1 Авторизация и активация (owed; вне M0 — но явно, не молча).**
- **user → Синапс.** Сейчас любой голос в салоне (пассажир, радио, голосовуха) = команда домашнему ПК. M0-допущение фиксируется явно: push-to-talk / привязка к доверенному устройству; wake-word + enrolled-voice — отдельный этап.
- **Синапс → Кора.** Канал моста и то, что Коре *позволено* исполнять, не описаны. Требование: mutual-auth + шифрование канала; authz-политика (allowlist / capability scope), принудительно **на стороне Коры**, независимо от промпта диспетчера.

**11.2 Подтверждение деструктива.** Механизм — Р-12 (§4): необратимые действия проходят голосовой read-back+подтверждение до исполнения. Разрывает компаундинг-цепочку: STT-ослышка (§7) → нет переспроса → необратимое действие → SPEAK подтверждает несанкционированное; за рулём глазами не проверить.

**11.3 Деградация и фолбэки (owed).** 3+ вендора (STT + TTS + LLM-диспетчер) на критпути одной реплики, ни одного фолбэка; за рулём на дёрганой сотовой обрыв WS посреди хода — норма, не край. Требования:
- пер-нога политика отказа: STT reconnect с реплеем аудио; **вторичный TTS авто-своп**; таймаут LLM → retry;
- **ошибки доходят до ушей, а не только до лога**: каждый класс отказа → голосовая фраза («связь потеряна», «не расслышал, повтори»); сейчас единственный сигнал — лог-алерт «статус без grounding» (§4), который пользователь не слышит;
- таблица стоимость/ход по всем вендорам до заморозки слотов (Fish — «уточнить», Прил. Б).

**11.4 Приватность (owed).** Сырой голос + транскрипты + содержание задач уходят в 5 сторонних облаков (Deepgram, ElevenLabs, AssemblyAI, OpenAI, Anthropic); retention / consent / redaction не заданы, журнал (§3) хранит полные транскрипты без лимита. Требуется: per-vendor no-train/retention-позиция, retention+redaction журнала, позиция по чувствительному контенту (в салоне вслух — имена, адреса, пароли).

**11.5 Секреты и стоимость (owed).** Ключи (Deepgram / TTS / LLM / Fish) — именованное хранилище + ротация, не «ключ уже есть» вразброс. Cost-cap на сессию/сутки с алертом: застрявший barge-in или доступ к неавторизованному микрофону жгут кредиты у 5 метред-вендоров. Таблица стоимость/ход — см. 11.3.

**Внесено вторым проходом разбора (2026-07-10):** #6 (§6 owed + §8 крит.4 на голосе), #7 (Р-13, §3/§7), #11 (§7 числовые гейты), #12 (§9 + §11.1), #13 (Прил. Г), #14 (§8 корпус/окно + §11.5). Открытых пунктов разбора не осталось — дальше боевые замеры и детальный дизайн owed-механизмов.

## Приложение Г — референсные WebSocket-реализации голосового слоя

Две рабочие реализации из нашего стека, покрывающие обе ноги каскада. Берутся как **референс протокола и паттернов**, не как готовый прод-код: оркестратор M0 — Pipecat (Python), а источники — TypeScript/React Native. По своему DESIGN `fishaudio-engine` портируется 1:1 на Python/Swift/RN.

**Источники:**
- **`fishaudio-engine`** — `github.com/Terobyte/fishaudio-engine` (**запинить на commit SHA — owed:** внешний репо дрейфует/исчезает) — платформо-агностичный WebSocket-движок Fish Audio TTS (Node 22+/Bun/Deno, msgpack, PCM16LE), с тестами. Чистый эталон: `common/websocket.ts` (обёртка сокета) + `tts/client.ts` (протокол, backpressure, abort).
- **AskME** (`~/Documents/trae_projects/AskME/AskME-Mobile/src/services/`) — боевые RN-сервисы: `deepgram-service.ts` (STT), `deepgram-streaming-service.ts` (Aura-2 TTS), `fish-audio-streaming-service.ts` (Fish TTS).

**Карта «источник → нога каскада»:**

| Файл | Нога | Endpoint | Аутентификация | Формат провода |
|---|---|---|---|---|
| `deepgram-service.ts` | **STT** | `wss://api.deepgram.com/v1/listen?encoding=linear16&sample_rate=16000&endpointing=300&interim_results=true` | `Sec-WebSocket-Protocol: ['token', <key>]` | вход PCM16 → выход JSON-транскрипты (interim + final) |
| `deepgram-streaming-service.ts` | TTS (Aura-2 — **REJECTED, только паттерн**) | `wss://api.deepgram.com/v1/speak?…&model=aura-2-*` | тот же subprotocol | control JSON (`Speak`/`Flush`/`Close`) + бинарный PCM16 |
| `fish-audio-streaming-service.ts` + `fishaudio-engine` | **TTS** | `wss://api.fish.audio/v1/tts/live?model=s1` | `Authorization: Bearer <key>` (header; RN 0.74+/undici) | **msgpack** обе стороны: `start`/`text`/`flush`/`stop` → `audio`/`finish`/`log`; PCM16LE@44.1k |

**Паттерны, которые забираем в M0:**
1. **Детекция конца реплики на STT** — `endpointing=300` + `interim_results=true` у Deepgram `/listen` снимает внешний VAD и напрямую закрывает риск turn-taking из §10. ⚠️ Это **классический** Deepgram-стриминг (`/listen`) — *другой продукт*, чем кандидат **Flux** (у Flux собственная модель конца реплики и свой API/параметры). Паттерн иллюстрирует подход; при выборе Flux endpoint и настройки сверять по его докам, не по `/listen`.
2. **Auth WebSocket без заголовков** — где клиент не может выставить header (браузер / старый RN), Deepgram использует subprotocol-трюк `['token', key]`; Fish идёт через `Bearer` header. На сервере Pipecat (Python) — обычные заголовки, оба варианта тривиальны.
3. **Artifact-free аудио** — msgpack + сырой PCM16LE, никаких codec-frame границ → нет кликов на стыках чанков (совпадает с §5: стриминг на каждом стыке). Правило: чанки не мутировать и не ре-буферизовать; потребитель владеет playback-буфером.
4. **Стриминговый интерфейс** — `async generator` чанков; у Fish есть форма `stream(asyncIterable<text>)` — аудио генерится параллельно токенам LLM (прямо ложится на «LLM стримит в TTS», §5).
5. **Barge-in и устойчивость** — `AbortSignal` прерывает поток и шлёт `stop` (нужно для перебивания реплики, §3/§10); backpressure по `maxBufferedChunks`; reconnect с exp-backoff; ping-keepalive.

**Несущие строки протокола (инлайн — на случай гниения ссылок):**
- **Fish `tts/live`** (msgpack обе стороны): клиент → `{event:"start", request:{format:"pcm", sample_rate:44100, reference_id, latency:"balanced", chunk_length:100}}`, затем N× `{event:"text", text}`, опц. `{event:"flush"}`, финал `{event:"stop"}`; сервер → `{event:"audio", audio:<bytes>}` ×N, `{event:"finish", reason}`, `{event:"log"}`. PCM16LE@44.1k: 2 байта = сэмпл, чанки конкатенируются без кликов, не мутировать. (`latency:"balanced"` — более быстрый TTFB, чем `normal`.)
- **Deepgram `/listen`** (STT): вход — бинарный PCM16; выход — JSON `{is_final, channel.alternatives[0].transcript}`; `endpointing=300` → событие конца реплики.

**Честные оговорки:**
- **Aura-2 TTS (`/v1/speak`) — только паттерн, не провайдер.** Спека уже отклонила Aura-2: русского нет (§3, Прил. Б). Берём схему auth/стриминга, не сам движок.
- Deepgram в AskME присутствует **и как STT, и как TTS** — это два разных файла и два endpoint (`/listen` vs `/speak`); в M0 полезен именно STT-путь.
- **Fish Audio → новый TTS-кандидат** (внесён в Прил. Б): русский у нас уже боевой (`tg.py`, голос Гаврилова), клиент artifact-free и готов. Решающая проверка — smoke-test русского ушами (§7) и замер TTFA. Это кандидат, не принятое решение — отдельного Р-решения пока нет.

**Ссылки на протоколы:** Deepgram STT streaming — `developers.deepgram.com/docs`; Fish Audio WebSocket — `docs.fish.audio/api-reference/endpoint/websocket/tts-live`.
