# Синапс · Спецификация M1 — рабочий инструмент (настоящая Кора + удалённый голос)

**Дата:** 2026-07-11 · **Статус:** дизайн, в работе · **Предшественник:** M0 — голосовой слой построен (§1) · **Глубокий рационал M0:** [`2026-07-08-synapse-m0-voice-cascade-design.md`](./2026-07-08-synapse-m0-voice-cascade-design.md) (архив — не переписываем, стоим на нём)

**Что меняется относительно M0.** M0 доказал голосовой слой: живой круг STT → каскад LLM → TTS с эшелонированной обороной от галлюцинаций диспетчера. Но «Кора» в нём — **полый in-memory стор** (никто не исполняет), доступ — **только `localhost`**, разговор — **pull-only** (спросил → ответил). M1 превращает демо в **инструмент, которым реально пользуются в машине**: настоящий Claude Code за мостом, доступ с телефона откуда угодно, и главное — рабочая модель **«кинул задачу → замолчал → тебя пингуют, когда есть апдейт»**.

## 2. M1 — рабочий инструмент

### 2.0 Целевой цикл (ядро — ради него всё)

```
1. 🟢 говоришь задачу  → диспетчер → submit_task → мост → НАСТОЯЩАЯ Кора начинает работу
2. 🔇 уходишь в тихий режим (микрофон выкл, сессия/контекст/задача ЖИВЫ)
3.     Кора пашет минуты/часы в фоне, шлёт события в мост
4. 📢 на завершении / вопросе → ПРОАКТИВНЫЙ ПУШ: ты слышишь (динамик вкл), хотя микрофон выкл
5. 🟢 надо ответить / дать новую задачу → размьютил, поговорил → снова 🔇
```

Это переворот относительно M0: там всё было **pull** (ты инициируешь каждый ход). Здесь исполнение **асинхронно и длинно**, а инициатива уведомления — у Коры.

**Модель состояний (🟢/🔇 + push) фиксирована; открыт только ЖЕСТ переключения** 🟢↔🔇 (tap / PTT / hotword — §2.3, решает Теро). Ядро строится и тестируется на абстрактных mute/unmute, **не завися** от выбора жеста (R3-крит: снимает противоречие §2.0↔§2.3).

### Скоуп: v1-ядро vs M1.1 (R3-YAGNI — режем, чтобы первая поездка состоялась к сроку)

| В v1 (несущее, «кодить в машине заработало») | Отложено в M1.1 (продакшн-твёрдость / удобства) |
|---|---|
| §2.1 хост-синглтон + настоящая Кора + маппинг событий + **рабочая директория** + лог tool-use Коры | §2.5 весь $-счётчик и $-деноминированный cost-cap (v1 — тупой backstop на Кору по call-count/времени) |
| §2.2 PWA + Tailscale (ред. 2026-07-12, было: cloudflared+Access) | §2.6 весь `/compact`+`/clear` и авто-компакт (v1 — рестарт хоста между поездками) |
| §2.4 push **только** `started/completed/failed` + вопрос Коры (E5); сериализация + grace — баг-превенция, не фича | §2.4 «вехи Коры» + переключатель A/B (v1 — хардкод A) |
| §2.7 «разрыв канала ≠ смерть задачи» + один текущий клиент | §2.7 launchd always-on + зомби-реконсиляция + welcome-back шаблон (v1 — хост руками перед поездкой) |
| §2.8 permission-callback Коры + **единый** echo-then-commit + визуал-tap для деструктива | §2.9 персист «Кора ждёт ответа» на диск (v1 — в памяти хоста) |
| §2.9 `answer_kora` в памяти хоста | voice-auth, TURN-self-host (если не потребуется), мультирепо |

Core-цикл §2.0 не показывает деньги/компакт нигде — они не нужны для «сказал задачу → Кора сделала → пуш». Режем без ущерба цели.

**Глоссарий (R3-крит, «сессия» перегружена):** **хост** — долгоживущий процесс (§2.1). **Кора-сессия** — сессия Claude Code SDK (контекст Коры). **Разговор-контекст** — `LLMContext` диспетчера. **Клиент** — WebRTC-подключение PWA. Мут / дисконнект / `/clear` трогают РАЗНЫЕ из них.

### 2.1 Настоящая Кора (Claude Agent SDK на Маке)

**Несущая предпосылка (R1-крит, BLOCKER): хост — процесс-синглтон.** Сегодня `run_session()` зовёт `build_pipeline(cfg)` на КАЖДЫЙ WebRTC-offer (`webrtc_server.py:47-48`), пересоздавая `TaskStore`, `ConfirmFlow`, `SpeakLedger`, `ArbiterPolicy` и связь с Корой; переживает разрыв только `state.json`. Пока эти объекты живут внутри коннекта, §2.4/§2.7 невозможны. Поэтому **первый слайс M1** — вынести долгоживущий **хост** (мост + сессия Коры + `TaskStore` + `SpeakLedger` + `ConfirmFlow` + счётчик) в процесс, который **переживает** WebRTC-коннекты; `run_session()` больше не строит их, а **подключает** транспорт к уже работающему хосту.

- **Кора = headless Claude Code через Claude Agent SDK** (Python), запущенный на домашнем Маке в этом хост-процессе, co-located с Синапсом.
- Хост держит сессию Коры; `submit_task(text)` → запрос в SDK → SDK стримит сообщения/tool-use/результаты → мост **маппит их в `KoraEvent`** (`task_started`, прогресс, `task_completed`/`task_failed`, + готовые SPEAK-тексты для критфактов).
- **Маппинг событий — allowlist, не fail-safe-в-critical (R1-крит).** M0 Р-15б гонит неизвестное событие в `critical` (unknown⇒громко). Для болтливого SDK-стрима это штормило бы `CRITICAL_WITHOUT_SPEAK`. Поэтому для источника «Agent SDK» инверсия: известные типы → явный класс по таблице; **неизвестное → `narratable`-тихо + лог**, не critical. Таблица маппинга — обязательный deliverable плана, строится ДО включения живого пути.
- Модель Коры — **сильная** (Opus/Sonnet), в отличие от дешёвого диспетчера. Два мозга: диспетчер #1 + Кора #2; третий не нужен.
- Заменяет полую `TaskStore`-заглушку. `FakeKora` остаётся для тестов и оффлайн-прогонов.
- **Рабочая директория Коры (R3-крит, нужна для §2.8):** v1 — **одна сконфигурированная workspace-директория** (путь к репо/проекту в конфиге), не `$HOME`. Permission-callback (§2.8) денит всё вне неё + секрет-пути. Мультирепо/произвольный путь — M1.1. Отсюда же ответ «что Кора умеет» = её workspace + allowlist инструментов.
- **Наблюдаемость (R3-крит):** сырой tool-use стрим Коры (инструменты/файлы/команды) пишется в журнал — иначе «жива, но зависла в цикле или делает не то» недиагностируемо (liveness Р-11 знает только жива/нет; `journal` сейчас логирует лишь 4 инструмента диспетчера).
- **Открытые вопросы реализации** (в план): persistent SDK-клиент vs сессия-на-задачу; permission-mode Коры (→ 2.8 — несущее для безопасности).

### 2.2 Удалённый доступ — «телефония по-простому»

- **PWA** (Progressive Web App: та же WebRTC-страница `/client` + файл-манифест → ставится иконкой на домашний экран телефона, открывается на весь экран как приложение; **не** нативное, без App Store/ревью) **+ Tailscale** (ред. 2026-07-12, было: именованный cloudflared-туннель — отвергнут: HTTP-туннель не несёт WebRTC-медиа (UDP), Access требовал домен). `tailscale serve` → `https://<mac>.<tailnet>.ts.net` = стабильный HTTPS-URL (secure context для getUserMedia), бесплатно, без покупки домена; Синапс+Кора на Маке, сервер виден **только** устройствам тейлнета.
- Работает по сотовой в машине; звук — WebRTC (Opus, браузерный AEC, barge-in) — и сигналинг, и медиа идут внутри WireGuard-меша Tailscale.
- **Twilio (звонок на номер) — вне v1**, опция позже: требует публичного сервера + помесячная плата + звук 8 кГц (хуже barge-in). Для «рулить кодингом голосом» app-по-дате лучше.
- **Риск:** NAT-травёрс между телефоном и Маком берёт на себя Tailscale (DERP-релей как фолбэк вместо self-host coturn) — проверить качество звука через DERP на реальной сотовой.

### 2.3 Управление микрофоном — три состояния

| Состояние | Микрофон (вход/STT) | Динамик (выход/TTS) | Сессия/контекст/задача |
|---|---|---|---|
| 🟢 **Активный** | вкл | вкл | живы |
| 🔇 **Тихий** («выключить, но не сбрасывать») | **выкл, 0 токенов STT** | **вкл** (пуши слышно) | **живы** |
| 🗑 **/clear** | — | — | контекст сброшен (→ 2.6) |

- **⚠️ Модель взаимодействия — ОТКРЫТОЕ решение (R2-крит, требует Теро).** В целевом цикле (§2.0) тумблер — не редкая операция, а **механизм каждого хода**: «🟢 сказать → 🔇» = **два прицельных тапа на ход**, каждый — глаза с дороги. Обоснование «tap лучше PTT, руки заняты» на деле описывает недостаток tap (PTT — одно удержание во время уже нужной реплики; tap — отдельное действие ДО и ПОСЛЕ). Кандидаты v1: **(A)** tap-тумблер (как есть); **(B)** PTT-удержание на время реплики; **(C)** голосовой one-turn open в 🔇 (лёгкий hotword открывает мик на один ход, авто-🔇 по end-of-turn) — но это граничит с wake-word (§4). Рекомендация: обсудить перед реализацией §2.3; дефолт-заглушка — A, но с барьером «два тапа/ход» это скорее демо-режим, чем «кодинг за рулём».
- Mute глушит **только вход**: локальный трек + STT не крутятся. Выход не трогается — **в этом вся суть проактивных пушей** (2.4).
- **Индикатор состояния + аудио/тактильный cue (R2-крит):** смена 🟢/🔇 даёт короткий тон/вибро — иначе за рулём не проверить «я точно замьючен» не глядя (промахнулся по тумблеру на кочке → приватная речь утекла в STT/контекст). Визуального индикатора мало.
- **Тихий режим ≠ /clear:** сессия, контекст диспетчера и трекинг задачи продолжают жить.

### 2.4 Проактивные уведомления (Кора → тебя, push) — НОВОЕ

M0 pull-only; M1 добавляет **push**: хост инициирует речь по событию Коры **без запроса пользователя**.

- **Триггеры v1 (хардкод режим «A — тихо»):** `task_completed` / `task_failed` + **вопрос Коры** (E5, → 2.9). Всё остальное — **молча**. *«Вехи, которые сама Кора пометила» + переключатель A/B — M1.1 (скоуп-таблица §2.0): требуют лимита/учёта, а ядру не нужны.*
- **Как звучит (v1 — только дословно):** пуш озвучивает **SPEAK Коры дословно**, минуя LLM диспетчера. «Готово / веха» без критданных — **детерминированный шаблон** (без LLM). **Лёгкая обёртка диспетчера в push-канале — вне v1** (R1-крит): она (а) требует разбудить диспетчер синтетическим ходом, что портит `note_user_turn` (double-key Р-16) или обходит `check_grounding` (Р-2); (б) даёт диспетчеру озвучить критфакт своими словами — прямой обход инварианта; (в) льёт полезную нагрузку Коры в LLM free-tier (приватность, → 2.8). Дословный SPEAK всё это обходит.
- **Со состояниями:** пуш звучит и в 🔇 (ради этого он и есть), и в 🟢 — но **не режет на полуслове**: нужен сигнал `user_speaking` из STT/VAD в хост (**сейчас его нет, R1-крит**), критичное — вперёд, прогресс — ждёт паузы.
- **Машинерия, которой НЕТ (R1-крит, правка ложного утверждения):** канал «Кора инициирует речь» надо построить с нуля. Сегодня `on_speak()` лишь кладёт в очередь `ArbiterPolicy`, а та дренируется **только когда через слинкованный pipecat-пайплайн идёт фрейм** (`arbiter.py:100-109`) — в 🔇 фреймов нет, SPEAK Коры **зависнет навсегда**. `monitor_forever()` ничего не дренирует. И `SpeakLedger.register_critical()` **на реальном пайплайне не вызывается** (только в console-runner) — сторож `critical⇒SPEAK` на прод-пути мёртв. M1 обязан: (1) путь инъекции TTS-фрейма **вне очереди-по-фрейму** (хост толкает синтезируемую речь в выход независимо от входящих фреймов); (2) `register_critical` на живом пути; (3) `user_speaking`-сигнал.
- **Сериализация озвучки (R2-крит):** два канала (фрейм-драйвенная очередь арбитра + новый push-инжектор) идут через **один владелец выхода** — иначе наложение речи или отодвинутый read-back. Один сериализованный порт TTS.
- **Grace на речь юзера (R2-крит):** «критичное вперёд» **не режет юзера на первом слове** — по `user_speaking` даём короткий cooldown после начала его реплики, только потом критичный push получает приоритет (иначе голос Коры влетает поверх формулировки на манёвре).
- **Лимит вех и осмысленный шаблон (R2-крит):** «вехи, которые Кора пометила» → **строго rate-limited и учтены в счётчике** (иначе дорогой tool-call на каждую веху мимо учёта, 2.5); v1 по умолчанию — только `started/completed/failed` + вопрос Коры. Шаблон «готово» **интерполирует текст задачи** (`render_state_template` сейчас отдаёт contentless «Задача завершена.» без `t.text` → гарантированный уточняющий ход через платный каскад).

### 2.5 Счётчик токенов и денег — M1.1 (отложено, R3-YAGNI)

> **v1:** только тупой backstop на трату Коры — переиспользовать call-count `CostCap` (M0) на вызовы Коры + лимит времени/числа tool-вызовов на задачу. Полная $-бухгалтерия ниже — дизайн для M1.1.

- **Чинит заодно «metrics не работает»:** включаем `enable_metrics` / `enable_usage_metrics` на `PipelineTask` (сейчас их нет → RTVI-панель пустая).
- **Два источника usage:** диспетчер (метрики pipecat на каждый LLM-вызов, **тегированные ступенью каскада**) + **Кора** (Agent SDK репортит usage per task). *Живой каскад сейчас — **2 платные ступени** (OpenRouter → Anthropic), обе `paid=True` (`services.py:34-50`); Google-direct-free ступени в коде нет (R2-крит) — не тегируем источник, которого каскад не производит; вернём как отдельную ступень, если поднимем Google-billing.*
- **Токены → $:** у **OpenRouter** точная стоимость прямо в ответе (`usage` / generation-endpoint); Anthropic — по прайс-мапе.
- **Показ в PWA:** итоги **сессия + сегодня**, сплит **диспетчер / Кора** и по провайдеру. Пер-ход детально — в журнал.
- Кормит уже спроектированный cost-cap (M0 §11.5) живыми числами.
- **Кора тоже под cost-cap (R1-крит):** сейчас `CostCap` обёрнут только на дешёвый каскад диспетчера; дорогая Кора ничем не капается. Агрегирующая структура (сессия/сегодня, сплит диспетчер/Кора/провайдер) — новая, в `TaskStore`/`journal` её нет.
- **cost-cap должен считать $, не попытки (R2-крит):** сейчас `record_paid_attempt()` (`services.py:67-78`) меряет **число вызовов** против `max_paid_calls_per_day` — ход Коры (Opus) и ход диспетчера (Haiku) весят одинаково «1». Чтобы $-числа §2.5 реально ужесточали гейт, cap → **$-деноминированный** (порог в деньгах, а не в вызовах), иначе счётчик и cap живут в разных единицах.

### 2.6 /compact + /clear на оба мозга — M1.1 (отложено, R3-YAGNI)

> **v1:** рестарт хост-процесса между поездками. За одну поездку при триггерах только `started/completed/failed/question` лог событий не рвёт контекст. Дизайн ниже — для M1.1.

Два **независимых** контекста:

| | `/clear` | `/compact` |
|---|---|---|
| **Диспетчер** (LLMContext разговора) | чистый разговор | саммари истории (1 вызов LLM), заменяет историю |
| **Кора** (Claude Code сессия) | свежая SDK-сессия | нативный компакт Claude Code через SDK |

- **Триггер:** кнопки в PWA (v1). Голосом — позже, **с confirm** (случайный `/clear` за рулём опасен).
- **Инвариант:** `/clear` диспетчера **не теряет задачу** — [СОСТОЯНИЕ] переинжектится мостом из `TaskStore`. Трекинг переживает сброс разговора.
- **`/clear` Коры при активной задаче** теряет контекст незавершённой работы → **требует confirm через визуальный диалог PWA** (кнопка + подтверждение на экране), **не** голосовой `ConfirmFlow` (R1-крит: тот структурно отклоняет `submit` при активной задаче, `confirm.py:135-140`, и подтверждает *текст задачи*, а не клиентское управляющее действие). Голосовой `/clear` — вне v1.
- **Компакт должен таргетить правильный контекст (R2-крит).** У диспетчера **два** источника роста: `self._history` (реплики разговора) И блок **[СОСТОЯНИЕ]** (`render_state()` — append-only лог событий Коры, `state.py:229-248`, никогда не тримится). В целевом сценарии «долгая поездка, юзер молчит, Кора пашет часами» история **мала**, а лог событий **пухнет на каждом ходе** — т.е. настоящий драйвер роста именно лог. Значит компакт/тримминг обязан жать **и** лог событий (свёртка старых событий в саммари/окно), не только историю разговора.
- **Авто-компакт диспетчера** по порогу длины. Дефолт — консервативный порог. **Не крадёт дефицитный RPM (R2-крит):** авто-компакт делит `breaker`/каскад с живыми ходами (RPM free≈5); фоновый триггер на пике съест слот дешёвой ступени и уронит следующий ход юзера на дорогой хвост → компакт идёт по **отдельному бюджету/в простое**, а не поверх активного диалога.

### 2.7 Живучесть (инструмент, а не демо)

- **Следствие §2.1 (хост-синглтон):** задача Коры живёт в хост-процессе **независимо** от WebRTC-канала; разрыв канала ≠ отмена. Отмена — только явный `request_cancel`.
- **⚠️ Изменение относительно M0:** сейчас `on_client_disconnected` → `task.cancel()` (`webrtc_server.py:56-60`) убивает задачу при закрытии вкладки. M1: дисконнект **рвёт только транспорт этого коннекта** — закрыть сокеты/трек ЭТОЙ сессии, чтобы не потекло (R1-крит: снять cancel, но **заменить** его teardown'ом транспорта, иначе вернётся утечка, которую этот код чинил), **не** трогая задачу хоста.
- **Один текущий клиент (R2-крит):** хост держит **ровно одно** активное голосовое подключение; новый коннект (реконнект / вторая вкладка) **вытесняет** старый (teardown старого транспорта), push всегда идёт в **текущий**. Иначе — дубль речи и неясно, кому пуш. `active_sessions` (`webrtc_server.py:85`) под always-on **чистится**, а не растёт вечно.
- **Reconnect:** обрыв дэйты в машине — норма. PWA авто-реконнект к туннелю с backoff (**с потолком интервала**, R2-крит — иначе в мёртвой зоне сотовой молчание не ограничено). На реконнекте: **ресинк [СОСТОЯНИЕ]** + короткий **детерминированный «с возвращением, задача “X”: <статус>»** без LLM (R2-крит: иначе первый естественный вопрос «что там?» идёт по самому медленному пути, TTFT 5–8.5с).
- **Офлайн критфакт не теряется (R2-крит, инвариант):** critical-SPEAK, всплывший **пока юзер был offline**, [СОСТОЯНИЕ] намеренно не несёт (редакция Р-15), а `CRITICAL_WITHOUT_SPEAK` — алерт оператору, не юзеру. Хост **удерживает недоставленные critical-SPEAK** и **переигрывает их дословно** на реконнекте. `critical ⇒ SPEAK` обязан дойти до **юзера**, не только до журнала.
- **Always-on:** хост-процесс (Синапс+Кора) под `launchd` — переживает ребут/краш (авто-рестарт). **Реконсиляция при рестарте (R2-крит):** задача в `state.json` со статусом RUNNING, но без живой SDK-сессии Коры → через liveness (Р-11) помечается `stale/unreachable` и **освобождает единственный слот**, иначе зомби-RUNNING (`state.py:165-169`) клинит его навсегда.
  - **⚠️ Живой репро 2026-07-11 (слайс 0, ждёт слайса 1):** задача, ушедшая в RUNNING без единого события Коры, имеет `last_event_ts is None` → `liveness()` возвращает **OK** (`state.py:220`), а не stale. Предложенный «через liveness → stale → освободить слот» **не сработает** для задачи, которая ни разу не получила события (в M0 таких — все, продюсера нет). Слайс-1-реконсиляция обязана ловить **и** этот кейс: загруженная нетерминальная задача без re-attach к живой Коре в пределах grace-окна на буте → терминализовать, а не полагаться только на liveness. Сейчас обходится хендлом `request_cancel` («отмени задачу» → `CANCEL_REQUESTED` ∉ блокирующего набора) или обнулением `state.json`.

### 2.8 Безопасность (трогаем реальный компьютер)

**Несущий принцип (R1-крит): настоящая граница безопасности — permission-callback Коры в Agent SDK, а НЕ голосовой классификатор диспетчера.** `confirm.py` гейтит только *формулировку задачи* до старта и слеп к tool-call'ам, которые Кора делает ВНУТРИ сессии; «почини баг» (не деструктивно) → сразу COMMITTED, а Кора внутри может `git push --force` / поставить пакет / прочитать `.env` — мимо двойного ключа. Поэтому:

- **Каждый tool-call Коры проходит через политику хоста** (Agent SDK `can_use_tool` / permission-mode): деструктив / выход за рабочую директорию / чтение секретов (`~/.ssh`, `.env`, `~/.aws`) / сетевой egress → **деним по политике или эскалация в голосовой confirm** (маршрутизируется как вопрос Коры, 2.9). Классификатор диспетчера — **вторичный** гейт на фразу, не первичная граница.
- **Prompt injection — в скоупе (R1-крит):** инструкции в содержимом файла/веба могут заставить Кору сделать деструктив без деструктивной голосовой фразы. Граница на исполнении tool-call'а (см. выше) ловит и это; классификатор диспетчера — нет.
- **Сетевая граница:** не голый shared-secret (утёкший URL+токен = RCE на Маке). v1 — **Tailscale** (ред. 2026-07-12, было: Cloudflare Access): сервер недостижим извне тейлнета, identity = членство устройства в тейлнете (WireGuard-ключи), URL сам по себе ничего не даёт; это же закрывает B16 (неаутентифицированный сигналинг).
- **Редакционная граница (R1-крит):** дорогой/приватный контент (реальный код, `.env`, ключи) в LLM-контекст диспетчера (free-tier, ToS допускает обучение) **не попадает**. Диспетчер видит только [СОСТОЯНИЕ] с редакцией critical (Р-15) и короткие флаги статуса; критфакты идут в **динамик** дословно, не в LLM.
- **Деструктив = усиленный гейт того же примитива echo-then-commit (R1+R3):** Р-16 фильтрует семантику, не личность (радио/пассажир «да» может подтвердить). Поэтому деструктив добавляет к общему echo-then-commit (ниже) **осознанный 🟢** (одноходовое push-приоткрытие микрофона его НЕ удовлетворяет) + **визуальный tap-подтверждение в PWA**, не только голос. Voice-auth (enrolled voice) — вне M1; остаточный риск — owed.
- **Кора под cost-cap** (2.5) — дорогие вызовы тоже капаются.
- **Единый примитив «echo-then-commit» (R3-YAGNI: один механизм, не три параллельные стейт-машины).** Перед КАЖДЫМ `submit_task` диспетчер **дёшево, детерминированно эхает распарсенную задачу** («Понял: “X”. Запускаю») — ловит мисхёрд и ложный end-of-turn (§3) до траты реальной Коры. Деструктив — тот же примитив с усиленным гейтом (выше). Р-16 double-key — деталь реализации этого примитива, не отдельная машина.

### 2.9 Диалог с Корой (E5 — вопрос Коры) — НОВОЕ

Claude Code часто переспрашивает. Сегодня у диспетчера **ровно 4 возможности** (`tools.py`, `prompt.py`), «передать ответ Коре» среди них **нет** (R1-крит) — на вопрос Коры диспетчер либо откажет («не умею»), либо вежливо ответит от себя, и ответ до Коры **не дойдёт**; Кора зависнет, а `get_task_status()` скажет «running, сигнала нет» — механически grounded, семантически ложно-обнадёживающе (новый канал confab-by-omission).

- **5-я возможность `answer_kora(text)`** + состояние моста **«Кора ждёт ответа»**. Пока оно взведено, следующая реплика юзера роутится в `answer_kora`, **не** в `submit_task`. Промпт → v4 (5-я возможность в закрытом списке).
- Вопрос Коры приходит как push (2.4), дословно. Ответ — verbatim в SDK, без переписывания диспетчером.
- В 🔇 вопрос слышно; чтобы ответить — юзер сам уходит в 🟢 (одноходовое приоткрытие микрофона **не** годится для ответа, влекущего действие, — см. 2.8 self-confirm).
- **Таймаут и выход (R2-крит):** «Кора ждёт ответа» имеет таймаут (по образцу `confirm_timeout_s`, `confirm.py:122`) — по истечении вопрос **повторяется пушем**, а не виснет молча навсегда (иначе не отличить «пашет» от «ждёт меня 20 минут»). Явный выход из состояния тоже есть.
- **Персистентность (R2-крит):** состояние «ждёт ответа» — **в схеме `state.json`** (`state.py:295-299` сейчас его не знает), иначе не переживёт реконнект/рестарт хоста и ответ уйдёт не туда.
- **Не взводить посреди хода юзера (R2-крит):** если вопрос Коры / permission-эскалация прилетает, когда юзер **уже говорит** (`user_speaking`), состояние не защёлкивается на текущей фразе — иначе хвост несвязанной реплики уйдёт Коре как ответ (`loop.py:50-61`).
- **«Повтори вопрос» (R2-крит):** в 🔇 вопрос виден текстом в PWA + кнопка replay-SPEAK; естественное «что ты спросила?» в 🟢 **не** должно уйти в `answer_kora` дословно — нужен явный re-ask, не буквальный ответ.

## 3. Edge cases

**Сеть / транспорт**
- Обрыв дэйты посреди работы → auto-reconnect; **Кора не теряет задачу** (2.7); ресинк [СОСТОЯНИЕ] на возврате.
- Туннель/сервер лёг → PWA показывает «отключено», ретраит с backoff; **не висит молча**.
- ICE через NAT на сотовой не встал → нужен TURN *(риск 2.2)*.

**Кора**
- Кора упала/зависла → liveness `stale/unreachable` говорит правду *(есть, Р-11)*.
- **Кора задаёт уточняющий вопрос (E5)** → пуш; ответ роутится через `answer_kora` (2.9), не `submit_task`. Без 5-й возможности это молчаливый confab-by-omission (R1-крит).
- Кора просит разрешение на инструмент → confirm / allowlist (2.8).
- Огромный вывод (диф/логи) → за рулём не прочитать → **саммари голосом**, критфакты через SPEAK.
- **Новая задача поверх активной** → v1 = **одна активная задача**; вторая → **отказ** («занят задачей “X”, скажи снова, когда освобожусь»), **без очереди** (R1-крит: `TaskStore` моделирует ровно одну задачу; очередь = объект, который `get_task_status()` не может заземлить → диспетчер выдумывал бы позицию → прямой удар по Р-2). Очередь — отложена (§4). `request_cancel` освобождает слот.
- **Ложный end-of-turn × «одна задача» (R2-крит):** шум/пауза в середине диктовки → Flux рано детектит конец хода → диспетчер сабмитит **пол-задачи** как активную → продолжение отбивается «занят задачей “X”», хотя «X» — оборванная фраза самого юзера. Смягчает **эхо-read-back** (§2.8: «Понял: “X”. Запускаю» до `submit_task` даёт шанс поправить), + лёгкий `request_cancel`-и-переговори.

**Голос / UX**
- 🔇 и ты говоришь → ничего (STT выкл); индикатор состояния виден.
- Пуш пока ты **активно говоришь** → критичное вперёд, прогресс ждёт паузы предложения (не режет).
- Пачка событий Коры подряд → **коалесцируем**, не строчим очередью в динамик.
- Вопрос Коры в 🔇 → услышал → affordance «ответить» / короткое приоткрытие микрофона на один ход.
- Barge-in поверх SPEAK в 🟢 → арбитр *(есть)*. **В 🔇 barge-in недоступен (STT выкл, R2-крит):** длинный SPEAK Коры не заткнуть голосом. v1: пуши в 🔇 держим **короткими** (саммари, не полный диф — уже выше) + кнопка «стоп» в PWA; голосовой «стоп» в 🔇 упирается в мик-модель (§2.3, открытое решение).
- Радио/пассажир говорит → тумблер + confirm деструктива фильтруют.

**Деньги / лимиты**
- Cost-cap посреди поездки → hard-stop платных, голосовая заготовка, статус шаблоном без LLM *(спроектировано M0 §11.5)*.
- Все ступени каскада легли → «связь с мозгом потеряна» через SPEAK *(есть)*.
- Счётчик без данных провайдера → last-known + флаг stale.

**Контекст**
- `/clear` диспетчера ≠ потеря задачи (инвариант 2.6).
- Длинная поездка → авто-порог компакта диспетчера (2.6).

## 5. Журнал критики (соннеты)

- `user_speaking`-сигнал отсутствует; Кора вне cost-cap; /clear Коры через визуал PWA — §2.4/2.5/2.6.

**Раунд 2 — операционка (живучесть/конкурентность · латентность/деньги · UX за рулём), 24 находки.** Внесено:
- **Мик-модель — открытое решение для Теро** (тумблер = 2 тапа/ход, обоснование «не PTT» описывает недостаток tap) + аудио/тактильный cue смены состояния — §2.3.

- **Офлайн критфакт-replay** (critical-SPEAK за offline невосстановим из [СОСТОЯНИЕ]) — хост удерживает и переигрывает; **мульти-коннект → один текущий клиент**; **зомби-RUNNING реконсиляция** при рестарте; welcome-back шаблон; backoff-потолок; cleanup `active_sessions` — §2.7.
- **«Кора ждёт ответа»:** таймаут+повтор, персист в `state.json`, не взводить посреди хода юзера, «повтори вопрос» — §2.9.
- **cost-cap считает попытки, не $** → $-деноминированный порог; **usage = 2 платные ступени**, Google-free источника в коде нет — §2.5.
- **Сериализация двух каналов озвучки**, лимит вех + учёт, шаблон «готово» с текстом задачи, grace на речь юзера — §2.4.
- **Эхо-read-back дорогих задач** (ловит мисхёрд и **false-EOT**) — §2.8, §3; **barge-in недоступен в 🔇** — §3.

**Раунд 3 — противник (полнота · YAGNI · качество спеки), 24 находки.** Внесено:
- **Скоуп-split v1/M1.1** (спека раздулась за R1/R2) — режем $-счётчик §2.5, compact/clear §2.6, launchd/зомби/welcome-back §2.7, A/B+вехи §2.4, персист §2.9 в M1.1 — блок «Скоуп» в §2.0.

- **Нет приёмочных гейтов** (M0 имел §7/§8) → новый §6 с DoD-таблицей, тест-стратегией (async FakeKora), owed-числами, миграцией M0→M1.
- **Три confirm-механизма → один** echo-then-commit, деструктив = усиленный гейт — §2.8.
- **Глоссарий** («сессия» перегружена) — §2.0.
