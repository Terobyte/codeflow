# Синапс UI/UX v2 — «клиент как Codex»: проекты, треды, стадийный флоу

Дата: 2026-07-13 (~01:30–02:00, брейншторм голос-в-чат с Теро; v2 после критики 12 находок — «переиспользование готового» было переоценено, исправлено решением «стадия = отдельный запуск Коры»).
Статус: **черновик v2, на ре-ревью Теро**.
Предыстория: M1 v1-ядро сдано (слайсы 0–5 + багхант + светофор/логи, `archive-2026-07-13.md`). Живой осмотр prebuilt-UI (скриншот 390×844) подтвердил: это девелоперская консоль pipecat (Transport-селектор, BOT AUDIO/VIDEO-панели, дебаг-тулбар), а не продукт; точка-светофор легла на кнопку Connect (зелёное на зелёном). Решение Теро: перерабатывать целиком, подход A (свой тонкий клиент сразу, бэк эволюционно, мелкими слайсами).

## 1. Требования (ответы Теро, зафиксированы в брейншторме 2026-07-13)

1. **Вход = дом как Codex**: список проектов и последних тредов + большая кнопка «🎙 Открыть агента». Один тап до голоса, но обзор первым.
2. **Ввод = голос + текст** везде: за рулём говоришь, дома/в шуме печатаешь. Бэку нужен текстовый ход в диспетчера мимо STT.
3. **«Открыть агента» без проекта** — универсальный вход: базаришь с диспетчером, называешь папку/проект голосом — он сам знает, какой тред создавать.
4. **Главный режим — не fire-and-forget, а цикл**: тривиальное («сделай txt») уходит Коре сразу, как сейчас; серьёзная работа — диспетчер СНАЧАЛА собирает контекст у Теро (много сырого голосового текста), сам задаёт уточняющие вопросы, и только когда собралось — превращает в запрос.
5. **Запрос уходит Коре только после добра Теро**: диспетчер предлагает свод («отправляю вот это — верно?»), Теро подтверждает или правит; можно форсировать («всё, отправляй»).
6. **Кора отвечает вопросами** — диспетчер задаёт их Теро ПО ОДНОМУ, туда-сюда (механизм = слайс-3 `answer_kora`, живой). Дальше ресёрчи, спека, план — файлами в проекте.
7. **Код пишется ТОЛЬКО после явного «ок, пиши код»** — жёсткий гейт.
8. **Тред = новый контекст**: работаешь над задачей — новая фича/багфикс/захотел освежить контекст → новый тред. Диспетчеру нужна гигиена контекста (чистка/компакт) — «а то он с ума сойдёт». (= спека M1 §2.6 растворяется в тредах.)
9. **Проекты добавляются из UI/голосом** (проект = папка на Маке).
10. **Выбор модели Коры**: Opus / Sonnet / Fable.
11. **Рассуждения Коры и общение диспетчер↔Кора открываются ВНУТРИ треда задачи** («бац нажал и виу») — не отдельная глобальная страница.
12. Убрать весь дебаг-мусор prebuilt.

## 2. Экраны (3, mobile-first PWA)

Стек: vanilla JS + тёмная тема в стиле `/client/logs` (прецедент в репо), ноль сборки, всё отдаётся своим сервером (tailnet-only). PWA-манифест/иконки слайса 5 переиспользуются. Prebuilt переезжает на `/client/dev` (отладка), основной `/client/` — наш.

### 2.1 Дом
- Шапка: «Синапс» · светофор-точка (семантика цветов без изменений: FAILED→красный prio, stale/awaiting/pending→жёлтый, иначе зелёный) · ⚙.
- Большая кнопка «🎙 Открыть агента» → тред без проекта (universal вход, требование 3).
- ПРОЕКТЫ: имя, число тредов/активность; «+ проект» (ввод пути; голосовое добавление — через confirm-читку, security ниже).
- ПОСЛЕДНИЕ ТРЕДЫ: бейдж стадии + статус, заголовок, проект, время.

### 2.2 Тред
- Шапка: ‹ назад · проект/заголовок · точка-статус · чип стадии (СБОР / ЗАПРОС / СПЕКА·ПЛАН / КОД / ГОТОВО·УПАЛ; при вопросе Коры — бейдж «❓», см. §3).
- **Единая хронологическая лента**: реплики Теро (транскрипт/текст), ответы диспетчера, события диспетчер↔Кора (запрос отправлен, вопрос Коры, ответ Теро, SPEAK), **сворачиваемые блоки рассуждений Коры** — по умолчанию строка «🧠 Кора: N шагов», тап разворачивает (контент = существующий display-only sink; ключевание по task_id/треду — новая мелкая логика, §4; XSS-политика textContent-only сохраняется).
- **Гейт-карточки в ленте** (интерактив):
  - «Запрос готов → [Отправить Коре] [Правки] · модель: Sonnet ▾ (Opus/Sonnet/Fable)» — модель SPEC_PLAN-запуска;
  - «План готов → скажи „пиши код“ или [Пиши код] · модель: ▾» — модель CODE-запуска (это отдельный запуск, §3).
- Низ: текст-поле + отправить + большая кнопка микрофона (жест v1 = текущий tap-латч; PTT/hotword — открытый вопрос §2.3 M1-спеки, решается живьём).

### 2.3 Настройки (минимум)
Модель Коры по умолчанию · ссылка на `/client/dev` (prebuilt) · версия/коммит сервера.

## 3. Стадийный флоу треда (конечный автомат)

**5 стадий** (не 6 — фикс несогласованности v1-черновика):

```
COLLECT ──(диспетчер предложил свод / «сформируй запрос»)──► PROPOSE
PROPOSE ──(добро Теро: «да»/кнопка)──► SPEC_PLAN ──(«ок, пиши код» — читка/кнопка)──► CODE ──► DONE/FAILED
   │ (правки → назад в COLLECT)           │ внутри запуска: вопросы Коры по одному (awaiting)
```

«Вопросы Коры» — НЕ отдельная стадия: это под-статус awaiting ВНУТРИ запуска Коры (существующий `set_awaiting`-флаг стора; светофор на нём уже жёлтый). Чип треда: СБОР / ЗАПРОС / СПЕКА·ПЛАН / КОД / ГОТОВО·УПАЛ + бейдж «❓ вопрос» при awaiting.

**Два РАЗНЫХ механизма вопросов** (в v1-черновике были склеены — это два кодовых пути):
- **COLLECT-вопросы задаёт диспетчер** — его собственный turn-цикл (промпт-логика). Кора не запущена; `AskUserQuestion` — инструмент Коры и без живого запуска физически не существует.
- **Вопросы Коры** — только внутри живого запуска: `AskUserQuestion` → парковка на future → голос/UI → `answer_kora` (слайс 3, live-доказан).

**Ключевое решение: стадия = отдельный запуск Коры.** SPEC_PLAN и CODE — ДВА запуска. Гейт-правила, cwd проекта и модель — launch-параметры `ClaudeAgentOptions` (`kora.py:440-465`), в живой сессии НЕ мутируются и раннер не перезапускается посреди задачи — ничего из механики слайсов 1–4 не ломается. Носитель контекста между запусками — план-файл в проекте (принцип tero-воркфлоу: канон в файлах, не в чате; SPEC_PLAN-запуск обязан оставить самодостаточный план). Бонусы: модель выбирается пер-стадию (спека на Sonnet, код на Opus — требование 10), смена гейт-режима и папки проекта — бесплатны.
*Альтернатива рассмотрена и отклонена (v1):* держать одну сессию открытой между стадиями (Кора сама спрашивает «писать код?» через AskUserQuestion-парковку) — сохраняет невербализованный контекст ресёрча, но сессия висит часами (`kora_deadline_s` тикает пока Теро думает, зомби-риск, залоченная модель). Парковка M1.1.

- **COLLECT**: Кора не запущена. Диспетчер копит контекст и активно переспрашивает (требование 4). Накопитель = тред-документ (журнал реплик + структурируемая выжимка).
- **PROPOSE**: свод зачитывается/показывается; отправка только после подтверждения (Р-16-паттерн двухфазности).
- **SPEC_PLAN** (запуск №1): гейт-режим `docs-only` — читать весь проект можно, мутирующие tool-calls только в whitelist-пути документации. **Это НОВАЯ логика в `_gate_decision`** (`kora.py:467-506` — сейчас гейт чисто контейнментный: секреты → workspace-клетка → Bash/egress; понятия стадии там нет). Честный объём — слайс UI-3, он не лёгкий.
- **CODE** (запуск №2, ТОЛЬКО по явному «пиши код»): гейт-режим `full` — клетка проекта + секрет-денилист (всегда), docs-ограничение снято. Задача запуска = «реализуй по плану `<файл>`».
- **Быстрый путь**: тривиальная задача → сразу PROPOSE (свод = сама задача) → CODE-запуск как сейчас. Классификатор «тривиальное vs серьёзное» = дешёвый диспетчер — скрытая точка отказа, учтена в Р2.

## 4. Данные и бэк (эволюция, синглтон сохранён)

- **Проект**: `projects.json` (id, имя, абсолютный путь). CRUD из UI; голосовое добавление = confirm-читка пути (открытие новой папки Коре — расширение security-границы, всегда явное). Гейт-клетка Коры = папка проекта, передаётся как **launch-параметр запуска** (cwd в `ClaudeAgentOptions` — задаётся на старте, живьём не меняется); `KORA_WORKSPACE_DIR` остаётся дефолтом «агента без проекта». Следствие: **привязка треда к проекту фиксируется ДО первого запуска Коры**; смена проекта в треде с запусками запрещена — новый тред. Секрет-денилист — поверх, всегда.
- **Тред ↔ текущая модель (маппинг зафиксирован)**: тред = НАДСТРОЙКА, не замена. Каждый запуск Коры внутри треда = задача текущей модели (task_id в TaskStore). Синглтон «одна активная задача» НЕ меняется; тред хранит свои task_id + ленту. Поэтому `answer_kora`, SPEAK-ledger и [СОСТОЯНИЕ]-инжект переиспользуются КАК ЕСТЬ — их пер-задачная семантика не тронута.
- **Тред**: персист-файл (id, project_id — может быть null («агент»), title — авто из первой реплики, stage, created/updated, история ленты, task_ids). Тред, начатый «агентом», привязывается к проекту, когда Теро называет его: STT неточен → диспетчер выбирает кандидата ТОЛЬКО из projects.json (LLM-матч допустим), но привязка — только после **confirm-читки имени проекта** (тот же паттерн, что деструктивные задачи §2.8 и добавление проекта); имя вне списка → предложение добавить проект (confirm). Механика привязки — слайс UI-3. Тред = граница контекста диспетчера: новый тред → чистый LLM-контекст + [СОСТОЯНИЕ]-выжимка из стора; длинный тред → компакт выжимкой (закрывает M1-спеку §2.6 без отдельной фичи).
- **Одна Кора = один исполняемый тред за раз** (v1): запуск из треда Б при исполняющемся А → вежливый отказ + бейдж «ждёт» (никакого авто-cancel А и авто-старта Б); смотреть Б можно всегда (read-only лента); освободить Кору — явный cancel в треде А. Параллельные треды/Коры = M2 (снос синглтона — отдельный этап).
- **Лента Коры per-тред**: display-only sink сейчас — ОДИН ring-буфер на хост (`kora_log_max`); ключевание записей по task_id (→ тред) — новая, хоть и мелкая, логика в sink (слайс UI-1), не буквальное переиспользование.
- **API** (Starlette-роуты до mount, паттерн устоялся): `GET/POST /api/projects` · `GET /api/threads?project=` · `POST /api/threads` · `GET /api/threads/{id}/feed` (лента, включая кора-шаги) · `POST /api/threads/{id}/message` (текстовый ход диспетчера — тот же turn-цикл, мимо STT) · `POST /api/threads/{id}/gate` (добро на запрос / пиши код — дублирует голосовые триггеры).
- **Миграция `/client/` без слома живых путей**: под `/client/*` живут НЕ только prebuilt-ассеты — живые роуты `/client/kora-status`, `/client/session-alive`, `/client/reconnect.js`, `/client/logs` (webrtc_server.py:298-352) и сигналинг `/start` + `/sessions/{id}/api/offer`. Переезжает ТОЛЬКО статика prebuilt: `app.mount("/client", …)` → `app.mount("/client/dev", …)`; живые роуты остаются на своих путях (reconnect.js/kora-status нужны prebuilt-обёртке на новом месте — проверить пути в патченном index). Наш index встаёт роутом `/client/` ДО mount. DoD слайса UI-1: prebuilt на `/client/dev` остаётся рабочим end-to-end (offer-хендшейк не сломан).
- **Модель Коры**: launch-параметр КАЖДОГО запуска (claude-opus-4-8 / claude-sonnet-5 / claude-fable-5) — выбирается на карточке PROPOSE (для SPEC_PLAN-запуска) и на карточке «Пиши код» (для CODE-запуска; дефолт = модель предыдущего запуска), общий дефолт в настройках. Пер-стадийный выбор — прямое следствие решения «стадия = отдельный запуск».
- **Голосовой транспорт в своём клиенте**: официальный pipecat JS SDK (client-js + small-webrtc transport), вендорится статикой. Коннект-логика становится нашей → закрывает парковку слайса 5 (умирает-после-3-ретраев, tap-to-connect).

## 5. Скоуп: слайсы v1 (каждый — живой продукт)

1. **UI-1 «каркас»**: свой `/client/` (дом + проекты из projects.json + добавление из UI + список тредов), тред-вью read-only с лентой Коры, светофор в шапке, **голос через свой клиент** (риск №1 — доказывается первым), prebuilt → `/client/dev`.
2. **UI-2 «текст»**: `POST message` в диспетчера, единая лента реплик в треде.
3. **UI-3 «стадии»** (самый тяжёлый слайс — честно): state-machine тредов, COLLECT-промпт диспетчера, гейт-карточки, **гейт-режим `docs-only`/`full` как launch-параметр + новая логика в `_gate_decision`** («без кода до CODE»), выбор модели пер-запуск, привязка треда к проекту (confirm-читка).
4. **UI-4 «гигиена»**: новый тред = чистый контекст, компакт длинного треда (§2.6), заголовки тредов.

**Не в v1**: параллельные Коры/треды (M2), режимы диспетчера (interview и др. — отдельный брейншторм), hotword/PTT (живой выбор жеста), пуш-уведомления.

## 6. Риски и открытые вопросы

- **Р1 (главный): голос в своём клиенте.** pipecat JS SDK в чистом виде без prebuilt — интеграцию доказываем в UI-1 до всего остального. Фолбэк: iframe/переход на prebuilt для голосовой части (уродливо, но страхует).
- **Р2: качество дешёвого диспетчера — ДВЕ точки отказа.** (а) COLLECT-структурирование сырого потока: выжимку копим детерминированно (журнал), «умную» часть эскалируем на Sonnet (эскалация в каскаде уже есть). (б) Классификатор «тривиальное vs серьёзное» (быстрый путь §3): недооценка → код без контекста, переоценка → interview на «сделай txt». Митигация: при сомнении классифицировать как серьёзное (цена — лишний confirm, не сломанный код) + PROPOSE-подтверждение стоит на обоих путях.
- **Р3: гейт-режим `docs-only` — новая логика, не переиспользование.** `_gate_decision` сегодня не знает стадий; whitelist docs-путей надо определить точно (читать весь проект можно, писать — только доки; куда попадают `plan.md`/`docs/`/`specs/` конкретного проекта). Точная форма — в план-фазе UI-3.
- **Р4: миграция `/client/` может сломать сигналинг/prebuilt** (пересекается с Р1): живые роуты и mount-порядок перечислены в §4; страховка — DoD UI-1 «prebuilt на /client/dev работает end-to-end».
- **О1**: жест микрофона (tap/PTT) — живьём. **О2 (решено для v1)**: очередь тредов = бейдж «ждёт», без авто-старта; конфликт «А исполняется, Б стартует» описан в §4 (отказ + явный cancel). **О3**: голосовое добавление проекта — дефолт v1 = UI-only, голосом позже (confirm-путь дороже, а добавление проекта — редкое действие у экрана).
