# Синапс UI/UX v2 — «клиент как Codex»: проекты, треды, стадийный флоу

Дата: 2026-07-13 (брейншторм голос-в-чат с Теро; v2 — после критики 12 находок «переиспользование переоценено»; v2.1 — фактология; v3 — после аудита 3 хантеров: 7 дизайн-пробелов чата + S1–S36 из `bugs.md`; **v3.1 — раунд 4: снапшот только для mode был половинчат → единый носитель RunSpec для cwd/gate_mode/model/thread_id**).
Статус: **черновик v3.1, на ре-ревью Теро**.
Предыстория: M1 v1-ядро сдано (слайсы 0–5 + багхант + светофор/логи, `archive-2026-07-13.md`). Живой осмотр prebuilt-UI (скриншот 390×844) подтвердил: это девелоперская консоль pipecat (Transport-селектор, BOT AUDIO/VIDEO-панели, дебаг-тулбар), а не продукт; точка-светофор легла на кнопку Connect (зелёное на зелёном). Решение Теро: перерабатывать целиком, подход A (свой тонкий клиент сразу, бэк эволюционно, мелкими слайсами).

**Терминология (S21):** «запуск» = один прогон Коры (= задача/task_id в TaskStore); «запрос» = свод, который тред несёт через стадии. Слово «задача» ниже употребляется только в смысле TaskStore.

## 1. Требования (ответы Теро, зафиксированы в брейншторме 2026-07-13)

1. **Вход = дом как Codex**: список проектов и последних тредов + большая кнопка «🎙 Открыть агента». Один тап до голоса, но обзор первым.
2. **Ввод = голос + текст** везде: за рулём говоришь, дома/в шуме печатаешь. Бэку нужен текстовый ход в диспетчера мимо STT.
3. **«Открыть агента» без проекта** — универсальный вход: базаришь с диспетчером, называешь папку/проект голосом — он сам знает, какой тред создавать. (Механика, S23: тред создаётся сразу при входе; «сам знает» = диспетчер потом ПРИВЯЗЫВАЕТ этот тред к проекту, §4.)
4. **Главный режим — не fire-and-forget, а цикл**: тривиальное («сделай txt») уходит Коре сразу, как сейчас; серьёзная работа — диспетчер СНАЧАЛА собирает контекст у Теро (много сырого голосового текста), сам задаёт уточняющие вопросы, и только когда собралось — превращает в запрос.
5. **Запрос уходит Коре только после добра Теро**: диспетчер предлагает свод («отправляю вот это — верно?»), Теро подтверждает или правит; можно форсировать («всё, отправляй»).
6. **Кора отвечает вопросами** — диспетчер задаёт их Теро ПО ОДНОМУ, туда-сюда (механизм = слайс-3 `answer_kora`, живой). Дальше ресёрчи, спека, план — файлами в проекте.
7. **Код пишется ТОЛЬКО после явного «ок, пиши код»** — жёсткий гейт (на быстром пути роль гейта играет явная PROPOSE-карточка «сразу код», §3/S1 — тихого fire-and-forget нет нигде).
8. **Тред = новый контекст**: работаешь над задачей — новая фича/багфикс/захотел освежить контекст → новый тред. Диспетчеру нужна гигиена контекста (чистка/компакт) — «а то он с ума сойдёт». (= спека M1 §2.6 растворяется в тредах.)
9. **Проекты добавляются из UI/голосом** (проект = папка на Маке; голосовое добавление — после v1, S11/О3).
10. **Выбор модели Коры**: Opus / Sonnet / Fable.
11. **Рассуждения Коры и общение диспетчер↔Кора открываются ВНУТРИ треда задачи** («бац нажал и виу») — не отдельная глобальная страница.
12. Убрать весь дебаг-мусор prebuilt.

## 2. Экраны (3, mobile-first PWA)

Стек: vanilla JS + тёмная тема в стиле `/client/logs` (прецедент в репо), ноль сборки, всё отдаётся своим сервером (tailnet-only). PWA-манифест/иконки слайса 5 переиспользуются НАШИМ index (судьба prebuilt — §4 «миграция»). Времена в UI — локальное время устройства; в персисте ISO-8601 (S33).

### 2.1 Дом
- Шапка: «Синапс» · светофор-точка — **семантика = существующий `_status_color` один-в-один** (S8: вкл. `UNREACHABLE→красный`, `CANCEL_REQUESTED→жёлтый`; спека НЕ переписывает таблицу цветов — источник истины код) · ⚙.
- Большая кнопка «🎙 Открыть агента» → тред без проекта (universal вход, требование 3).
- ПРОЕКТЫ: имя, число тредов/активность; «+ проект» — ввод пути, **только UI в v1** (S11; голосовое добавление — после v1, механизм = confirm-читка).
- ПОСЛЕДНИЕ ТРЕДЫ: чип стадии + бейдж исхода последнего запуска, заголовок, проект, время.

### 2.2 Тред
- Шапка: ‹ назад · проект/заголовок · точка-статус · чип стадии (СБОР / ЗАПРОС / СПЕКА·ПЛАН / КОД / ГОТОВО — 5 стадий, §3) + бейджи поверх стадии: «❓» при awaiting (деривация S16: awaiting живёт в синглтон-сторе → бейдж возможен ТОЛЬКО у треда активного запуска), «✖ упал» / «⏹ отменён» — исход последнего запуска (FAILED — не стадия, §3/S2/S15).
- **Единая хронологическая лента**: реплики Теро (транскрипт/текст), ответы диспетчера, события диспетчер↔Кора (запрос отправлен, вопрос Коры, ответ Теро, SPEAK), $-строка после result-шага (S20 — в DoD UI-2), **сворачиваемые блоки рассуждений Коры** — по умолчанию строка «🧠 Кора: N шагов», тап разворачивает (персист и ключевание — §4/S3; XSS-политика textContent-only сохраняется).
- **Гейт-карточки в ленте** (интерактив; действия — §4 API):
  - стадийный путь: «Запрос готов → [Отправить Коре] [Правки] · модель: Sonnet ▾» — модель SPEC_PLAN-запуска;
  - быстрый путь (S1): «Запрос готов → [Отправить Коре — СРАЗУ КОД] [Правки] · модель: ▾» — модель CODE-запуска, двухфазный тап;
  - «План готов → скажи „пиши код“ или [Пиши код] [Правки] · модель: ▾» (S2: [Правки] есть; S4: [Пиши код] — двухфазный тап «точно пишем код в <проект>?» — HTTP-эквивалент Р-16-читки);
  - после упавшего/отменённого запуска: «[Повторить] [Правки → СБОР]» (S2).
- Низ: текст-поле + отправить + большая кнопка микрофона (жест v1 = текущий tap-латч; PTT/hotword — открытый вопрос §2.3 M1-спеки, решается живьём). При awaiting текстовый ход = ответ Коре: диспетчер зовёт `answer_kora` (S17).
- Заголовок треда: авто из первой реплики; переименование — UI-4 (S30).

### 2.3 Настройки (минимум)
Модель Коры по умолчанию · ссылка на `/client/dev` (prebuilt) · версия/коммит сервера.

## 3. Стадийный флоу треда (конечный автомат)

**5 стадий треда** (S15/S22 — честный подсчёт): COLLECT → PROPOSE → SPEC_PLAN → CODE → DONE. **FAILED — не стадия треда, а исход запуска** (см. ниже). Кору запускают только SPEC_PLAN и CODE; COLLECT и PROPOSE — чисто диспетчерские, без Коры (S22).

```
COLLECT ──(свод готов)──► PROPOSE ──(добро Теро)──► SPEC_PLAN ──(«пиши код» + план-файл есть)──► CODE ──(успех)──► DONE
   ▲                         │ [Правки]                 │ [Правки]                                  │
   └─────────────────────────┴──────────◄───────────────┘                                           │
Быстрый путь: COLLECT ──► PROPOSE («сразу код», свод = задание) ────────────────────────────────► CODE
Упавший/отменённый запуск: стадия НЕ меняется → карточка [Повторить] / [Правки → COLLECT]
```

**Исходы запусков (S2):** стадия описывает прогресс треда и двигается только вперёд по гейтам (или назад по [Правки]). Исход запуска — успех / упал / дедлайн / отменён — ортогональный статус последнего task_id (бейдж в шапке + светофор; DONE и FAILED в коде/enum раздельны — S15). Упал или отменён → тред остаётся в своей стадии с карточкой «[Повторить] [Правки]»; повтор = новый запуск той же стадии. DONE треда = успешный CODE-запуск. Тупиков в автомате нет.

**Гейт SPEC_PLAN→CODE — механизм, не модальность (S2):** кнопка «Пиши код» активна только когда SPEC_PLAN-запуск завершился успешно И план-файл существует в docs-whitelist (проверка наличия файла бэком; конвенция пути план-файла — план-фаза UI-3). Файла нет → кнопка неактивна, подсказка «повтори СПЕКА·ПЛАН».

**Быстрый путь (S1):** SPEC_PLAN пропускается, план-файла нет → задача CODE-запуска = сам свод. Требование 7 удовлетворяет PROPOSE-карточка с явной пометкой «СРАЗУ КОД», селектором модели CODE-запуска и двухфазным тапом. Классификатор «тривиальное vs серьёзное» = дешёвый диспетчер — точка отказа, учтена в Р2; при сомнении — серьёзный путь.

**Два РАЗНЫХ механизма вопросов** (два кодовых пути):
- **COLLECT-вопросы задаёт диспетчер** — его собственный turn-цикл (промпт-логика). Кора не запущена; `AskUserQuestion` — инструмент Коры и без живого запуска физически не существует.
- **Вопросы Коры** — только внутри живого запуска: `AskUserQuestion` → парковка на future → голос/UI → `answer_kora` (слайс 3, live-доказан).

**Ключевое решение: стадия с Корой = отдельный запуск.** SPEC_PLAN и CODE — ДВА запуска. Гейт-режим, cwd проекта и модель — launch-параметры, которые несёт **RunSpec** (ниже; сегодня `_build_options` берёт cwd/model из ОБЩЕГО конфига — `_workspace()`/`kora_model`, kora.py:451/460 — это меняется), в живой сессии не мутируются, раннер не перезапускается посреди запуска — механика слайсов 1–4 не трогается. Носитель контекста между запусками — план-файл в проекте (канон в файлах, не в чате; самодостаточность плана обеспечивает гейт-проверка наличия файла выше). Бонусы: модель пер-стадию (спека на Sonnet, код на Opus — требование 10), смена гейт-режима и папки — бесплатны.
*Альтернатива отклонена (v1):* держать одну сессию открытой между стадиями (AskUserQuestion-парковка «писать код?») — сессия висит часами (`kora_deadline_s` тикает, зомби-риск, залоченная модель). Парковка M1.1.

**Носитель launch-параметров — RunSpec (чат-находка 1, дочинено раундом 4) — решено:** `start()`/`_run()` получают, помимо `(task_id, text)`, объект **RunSpec** = `{thread_id, project_root | null, gate_mode: docs_only|full, model}` (сегодня сигнатура — только `(task_id, text)`, kora.py:337/373; расширение обязательно — без thread_id невозможен и маппинг task_id↔тред из UI-1). В начале `_run`, ДО создания клиента, RunSpec атомарно кладётся в per-run снапшот на инстансе раннера. Дальше **один источник истины на всё**: `_build_options` берёт cwd из `RunSpec.project_root` (fallback `_workspace()` при null) и model из `RunSpec.model`; `_pretool_hook`/`_gate_decision` читают ТОТ ЖЕ снапшот — рассинхрон «Кора пишет в один путь, гейт проверяет другой» (ложные deny или дыра) невозможен по построению. Прокинуть параметры в гейт через `ClaudeAgentOptions` нельзя — PreToolUse-хук не получает кастомного payload, он замыкает `self`. Защита от «чужого» запуска — тот же identity-guard-паттерн (владелец = task_id), что у `_pending_answer`; сброс — в структурном finally с проверкой владельца.

**Межзапускный reset (чат-находка 2) — решено:** начало каждого `_run` = чистые per-run слоты (gate-снапшот; `_pending_answer`, не принадлежащий этому запуску, не читается — identity-guard). Cancel-пути уже чистятся identity-guard'ом `_handle_question` (слайс 3). Обязательный тест UI-3: awaiting из SPEC_PLAN-запуска не протекает в CODE-запуск.

**Single-flight гейта (S6) — требование:** gate-действия на тред сериализуются (per-thread lock); повторный gate при живом запуске → 409, идемпотентно. UI-путь НИКОГДА не вызывает supersede-`start()` поверх живого рана (`KoraRunner.start` сегодня cancel'ит предыдущий — двойной тап убил бы запуск посреди Write); освобождение — только явный cancel.

- **COLLECT**: Кора не запущена. Диспетчер копит контекст и активно переспрашивает (требование 4). Накопитель = тред-документ (журнал реплик + структурируемая выжимка).
- **PROPOSE**: свод зачитывается/показывается; отправка только после подтверждения (Р-16-паттерн двухфазности).
- **SPEC_PLAN** (запуск №1): гейт-режим `docs_only` — читать весь проект можно, мутирующие tool-calls только в whitelist-пути документации. **Новая логика в `_gate_decision`** (`kora.py:467-506` — сейчас гейт чисто контейнментный, ветвится по типу инструмента (S35); понятия стадии нет). Честный объём — слайс UI-3.
- **CODE** (запуск №2, только по явному «пиши код» / «сразу код»): гейт-режим `full` — клетка проекта + секрет-денилист (всегда), docs-ограничение снято. Задача запуска = «реализуй по плану `<файл>`» (стадийный путь) или сам свод (быстрый путь, S1).

## 4. Данные и бэк (эволюция, синглтон сохранён)

- **Проект**: `projects.json` (id, имя, абсолютный путь). CRUD из UI (голосовое добавление — после v1, S11). **Валидация пути при добавлении (S12):** существующая директория; реджект `/`, `$HOME` целиком и системных корней (`/System`, `/Library`, `/usr`, `/etc`); секрет-денилист расширяется — как минимум `~/.config`, `~/Library/Keychains`, шелл-конфиги (`.zshrc`, `.bashrc`, `.bash_profile`, `.profile` — запись в них = persistence), `~/.gnupg` (точный список — план-фаза UI-1). `projects.json` пишется атомарно (tmp+rename) под asyncio-lock — UI и голос не гоняются (S28). Гейт-клетка Коры = папка проекта как **launch-параметр запуска** (едет в `RunSpec.project_root`, §3 — cwd И гейт читают один снапшот). Явно (раунд 4): **проектный тред → cwd = гейт-клетка = папка проекта; project-less тред → `RunSpec.project_root = null` → дефолт-воркспейс `KORA_WORKSPACE_DIR`** (разделяемый всеми «агент»-тредами, принято в v1 — S19; per-thread подпапки — M1.1). Сегодняшний `_workspace()` (kora.py:424-426, один путь на инстанс из конфига) остаётся только fallback'ом для null. Привязка треда к проекту фиксируется ДО первого запуска Коры; смена проекта в треде с запусками запрещена — новый тред.
- **Тред ↔ текущая модель (маппинг зафиксирован)**: тред = НАДСТРОЙКА. Каждый запуск Коры внутри треда = задача текущей модели (task_id в TaskStore). Синглтон «одна активная задача» НЕ меняется; тред хранит свои task_id + ленту. Поэтому `answer_kora`, SpeakLedger и [СОСТОЯНИЕ]-инжект переиспользуются КАК ЕСТЬ. **«Исполняемый тред» (S18) = тред с ЖИВЫМ запуском Коры**; COLLECT/PROPOSE-треды синглтон не занимают, их можно вести параллельно с чужим запуском. Текущий исполняемый тред — производная: тред, владеющий task_id активной задачи (маппинг появляется в UI-1 — он же нужен ленте; чат-находка 3).
- **Тред**: персист-файл (id, project_id — может быть null, title — авто из первой реплики (rename — UI-4, S30), stage, исход последнего запуска, created/updated, история ленты, task_ids). **Надёжность ленты (S3):** кора-шаги доезжают до персиста НЕ через ring-буфер — `log_sink` получает второго потребителя: append записей (по task_id → тред) в файл истории треда; ring-буфер остаётся горячим кэшем live-стрима. Лимит файла истории — конфиг `thread_feed_max` (аналог `kora_log_max`, S32). Рестарт хоста ленту не теряет. Писатель — слайс UI-1.
- **Привязка треда к проекту голосом**: STT неточен → диспетчер выбирает кандидата ТОЛЬКО из projects.json (LLM-матч допустим), привязка — после **confirm-читки имени проекта** (паттерн §2.8); имя вне списка → предложение добавить проект (в v1 — отсылка к UI, S11). Механика — UI-3.
- **Гигиена контекста / компакт (S10):** тред = граница контекста диспетчера: новый тред → чистый LLM-контекст + [СОСТОЯНИЕ]-выжимка. Компакт длинного треда: LLM-выжимка строится ТОЛЬКО по диспетчерскому журналу (реплики + события); **сырые кора-шаги в LLM-контекст диспетчера не попадают никогда** (NO-EXFIL, `kora.py:205-215`) — они display-only и остаются в файле истории. Авто-компакт с уведомлением в ленте; необратимого удаления нет (сырая история в файле). Закрывает M1-спеку §2.6.
- **Архив/удаление (S31):** v1 — архив треда (скрыт из списков, файл остаётся); удаление проекта отвязывает треды (project_id → null, пометка «проект удалён»). Полного удаления тредов в v1 нет.
- **Одна Кора = один исполняемый тред за раз** (v1): gate-запуск из треда Б при живом запуске в А → вежливый отказ + бейдж «ждёт» (авто-cancel А и авто-старта Б нет); смотреть/вести COLLECT в Б можно всегда; освободить — явный cancel в А. Параллельные треды/Коры = M2.
- **Зомби-реконсиляция на буте — втянута в v2 (S13):** сегодня крэш сервера посреди RUNNING восстанавливает RUNNING из state.json как есть, liveness отвечает OK, `has_active_task()` режет любой submit → ВСЕ треды заблокированы навсегда. Мини-фикс в UI-1: на буте задача в статусе RUNNING (живого рана после рестарта не существует по определению) → FAILED с причиной «сервер перезапускался». Без этого очередь «вежливый отказ» стоит на вранье.
- **API** (FastAPI-роуты поверх Starlette, регистрация до mount): `GET/POST /api/projects` · `GET /api/threads?project=` · `POST /api/threads` · `GET /api/threads/{id}/feed` · `POST /api/threads/{id}/message` · `POST /api/threads/{id}/gate`.
  - **gate (S5):** body `{action: "send_to_kora" | "write_code" | "revise", model?: "<id>", confirm?: true}`. `send_to_kora` — добро на SPEC_PLAN-запуск (или CODE на быстром пути — тогда `confirm` обязателен); `write_code` — CODE-запуск (confirm обязателен); `revise` — стадия назад в COLLECT ([Правки] ходит через gate, не message). `model` валидируется allowlist'ом из трёх id (S34) — невалидное → 400, в `ClaudeAgentOptions` сырым не уходит.
  - **Анти-CSRF (S4):** tailnet — сетевая граница, не браузерная; auto-submit `<form>` с любой открытой страницы дёргал бы gate. Все мутирующие `/api/*` требуют `Content-Type: application/json` (HTML-форма не может) + проверку Origin/Referer против собственного хоста. Плюс двухфазный confirm на `write_code`/«сразу код» (выше). Честно: tool перехода стадий в `dispatcher/tools.py` не существует — gate-контур НОВЫЙ (UI-3), «дублирует голосовые триггеры» = голосовые эквиваленты добавляются туда же.
  - **message (S7):** «тот же turn-цикл» = та же функция диспетчера, но `_current_turn_id`/дедуп сегодня НЕ канал-scoped — одновременный voice-ход и HTTP-ход перетёрли бы друг друга. Требование: одна очередь ходов на хост, HTTP-ход сериализуется с голосовым (новая мелкая логика UI-2). При awaiting message-ход = ответ Коре через `answer_kora` (S17).
- **Миграция `/client/` — упрощена решением v3 (S14/S25/S27, чат-находка 4): prebuilt на `/client/dev` = НЕПАТЧЕННЫЙ dist.** Инжекты слайса 5 (manifest/иконки/`reconnect.js`/`status-widget.js`) НЕ переезжают — PWA-обёртка и реконнект становятся обязанностью нашего index; непатченный prebuilt-index ссылается только на собственные `./assets/*`, которые резолвятся внутри mount'а — относительным ссылкам ломаться не на чем, манифест на dev-странице отсутствует (S25), фолбэк Р1 опирается на чистую статику (S27). Synapse-роуты остаются на `/client/*`: PWA-статика (`manifest.webmanifest`, `icon-192/512`, `apple-touch-icon`), `reconnect.js`, `status-widget.js`, API (`session-alive`, `kora-status`, `kora-log`), `logs`; `/client/` + `/client/index.html` начинают отдавать НАШ клиент. Вне `/client` и вне миграции (S36): `/start`, `POST/PATCH /sessions/{id}/api/offer`, `POST/PATCH /api/offer`, `GET /` (редирект). **Порядок регистрации (S24):** точные роуты → mount `/client/dev` → любые более общие mount'ы ПОСЛЕ. DoD UI-1: тест «точный роут И mount резолвятся оба»; prebuilt-консоль на `/client/dev` работает end-to-end (offer-хендшейк); manifest/иконки резолвятся с нашего index.
- **Модель Коры**: launch-параметр КАЖДОГО запуска (`claude-opus-4-8` / `claude-sonnet-5` / `claude-fable-5`; сервер-сайд allowlist, S34), едет в `RunSpec.model` (§3; конфиговый `kora_model` — дефолт) — выбирается на карточках (§2.2), дефолт CODE-запуска = модель предыдущего запуска, общий дефолт в настройках.
- **Голосовой транспорт в своём клиенте**: официальный pipecat JS SDK (client-js + small-webrtc transport), вендорится статикой. **Вендоринг (S26):** пин версии + лицензия в vendor-папке; проверка «ноль сборки» — бандл без bare-specifier-импортов (иначе берём собранный IIFE/ESM-бандл релиза) — входит в Р1-пробу UI-1. Коннект-логика становится нашей → закрывает парковку слайса 5 (умирает-после-3-ретраев, tap-to-connect).

## 6. Риски и открытые вопросы

- **Р1 (главный): голос в своём клиенте.** pipecat JS SDK в чистом виде без prebuilt — интеграцию доказываем в UI-1 до всего остального. Подпункт (S26): вендоринг может потребовать собранный бандл (bare-specifiers ломают «ноль сборки»). Фолбэк: переход на `/client/dev` (непатченный prebuilt — чистая статика, надёжен, S27) для голосовой части.
- **Р2: качество и стоимость дешёвого диспетчера — ТРИ точки отказа.** (а) COLLECT-структурирование сырого потока: выжимку копим детерминированно (журнал), «умную» часть эскалируем на Sonnet. (б) Классификатор «тривиальное vs серьёзное»: при сомнении — серьёзный путь (цена — лишний confirm, не сломанный код); PROPOSE-подтверждение стоит на обоих путях. (в) Объём (S29): активное переспрашивание жжёт платный каскад/RPM — вопросы пачкой в одном ходе, лимит уточнений на тред.
- **Р3: гейт-режим `docs_only` — новая логика.** Носитель решён (§3: RunSpec → per-run снапшот, один источник для cwd и гейта); осталось определить точный whitelist docs-путей (читать весь проект, писать — только доки; куда попадают `plan.md`/`docs/`/`specs/` проекта) и конвенцию пути план-файла для гейт-проверки — план-фаза UI-3.
- **Р4: миграция `/client/`** — снижен решением v3 (непатченный dist, ничего не патчим на новом месте); остаточный риск — порядок регистрации (S24, закрыт тестом DoD UI-1).
- **О1**: жест микрофона (tap/PTT) — живьём. **О2 (решено v1)**: очередь тредов = бейдж «ждёт», без авто-старта; конфликт А/Б — §4. **О3 (решено v1, согласовано везде — S11)**: добавление проекта только из UI.
